# オーケストレーション — 同一ホスト内のモデル階層運用

`routing.md` が「どのAIツールに投げるか」を決めるのに対し、本書は**同一ツールの内側**で
「どのモデルにやらせるか」を決める。委譲ラダーと検証規律はホスト共通。Claude Codeでは
既定挙動（詳細は `../roles/claude-code.md`）。Cursor 対話レーンでの載せ方は §7。

## 1. 指揮者の定義（モデル名に依存させない）

- **指揮者（オーケストレーター）= その時点で使える最上位モデル**
- モデル名は導入時点で変わる。本書では「最上位モデル」「中位モデル」「軽量モデル」という
  **相対階層**でのみ語る。固有名詞（バージョン名）はこのファイルに書かない
- 指揮者の仕事は**設計・委譲・検証・統合・提案**。物量作業を本人がやらないことで
  トークン対パフォーマンスを最大化する

## 2. 委譲ラダー（タスク開始時にまず設計する）

タスク開始時に「何をどのモデルでやるか」を先に決めてから着手する（着手しながら都度判断しない）。

| 作業 | 担当 | 判断の目安 |
|---|---|---|
| 軽い質問・即答・1〜2ファイルの読み取り | 指揮者が直接 | 委譲しない（サブエージェントは文脈再構築コストで逆に高くつく） |
| 大量・機械的処理（多数ファイルの読込・変換・棚卸し） | 軽量モデルのサブエージェント | 読むファイル3つ以上、または機械的反復 |
| 中規模の独立サブタスク（監査・調査・ドラフト・実装） | 中位モデルのサブエージェント | 独立していれば**並列起動**（3〜4体が目安） |
| 統合・検証・最終提案・ユーザー向け報告 | 指揮者本人 | 委譲しない |

- 独立サブタスクは並列・バックグラウンドで起動し、完了通知で回収する（待ちでブロックしない）
- 委譲プロンプトの型: **読み取り専用と明記 / 担当ファイルを列挙 / チェック観点を番号付きで /
  出力形式を固定 / 「問題なし」も明記させる**。この型で複数体の報告がそのまま突合可能になる

## 3. 検証規律（サブエージェント報告の扱い）

- 報告は**採用前に指揮者が一次資料で裏取り**する。重要度「高」の指摘と「事実誤認」系の指摘は必須
- 下位モデルは知識差による**誤検知**を出す。**修正案自体が間違っている**こともある。
  「指摘が正しいこと」と「修正案が正しいこと」は別々に検証する
- クラスタ分割で委譲すると、**分割の境界をまたぐ不整合は誰も見ない**。横断不変条件
  （マージ条件・ラベル名・verdict語彙・バージョン表記など）は、最後に指揮者がgrepで全数確認する
- 観点を明示して渡した項目はほぼ漏れなく検査される。渡していない観点は指揮者が自分で見るか、
  次回から観点に追加する
- 適用後の再チェックは2系統に分けると精度が上がる: ①横断不変条件の全数grep担当、
  ②編集起因の新規矛盾をfresh-contextで通読する担当。自分の編集は自分では見えないため、
  検証は必ず別コンテキストに出す

## 4. 指揮者本人の作業と承認ゲート

- **事前承認が必要なのは、不可逆4カテゴリに触れる作業と、正本（ルートの`AGENTS.md`・`CLAUDE.md`・基準ファイル）の
  変更のみ**。それ以外の状態変更（ファイル編集・作成・削除・設定変更・外部送信等）は指揮者が実行してよく、
  出力契約で事後報告する。読み取り・調査・回答・自身のメモリ管理はそもそも承認の対象外
  （この境界の正本はルートの`AGENTS.md`承認節。差分があればそちらが勝つ）
- 事前承認が必要な作業も、承認は作業単位ではなく**バッチ単位**でよい（「発見N件を適用してよいか」を1回で束ねる）

## 5. 上位・下位モデル両対応の原則（このファイル自体の設計方針）

- 世代の異なる指揮者の差は「できる/できない」ではなく**委譲の自発性**。下位世代ほど明示しないと
  委譲が減り、上位世代は明示されても劣化しない → **委譲指示は明示側に倒す**のが両対応の正解
- ただし手順の細かい列挙は避け、**判断基準だけを短く**書く（過度に規範的な指示は上位モデルの品質を下げる）
- 非最上位モデルが指揮者を担う場合の追加注意: §3の検証をより厳格に。長時間実行では、進捗報告を
  必ずツール実行結果と突き合わせてから出す（根拠のない完了報告をしない）

## 6. グローバル既定化スニペット（ユーザーレベル設定へコピー）

リポジトリ配布ではAGENTS.md経由で効くが、**全環境で既定にする**には、使っているツールの
ユーザーレベル設定（Claude Codeなら `~/.claude/CLAUDE.md` 相当）に以下を貼る:

```markdown
## オーケストレーション既定
タスク開始時に「何をどのモデルでやるか」を先に設計する。大量・機械的処理は軽量モデル、
中規模の独立サブタスクは中位モデルのサブエージェントへ委譲する（独立なら並列起動）。
1〜2ファイルで即答できる軽作業は本体が直接。サブエージェントの報告は採用前に一次資料で検証する。
事前承認が必要なのは不可逆4カテゴリに触れる作業と正本（AGENTS.md・CLAUDE.md・基準ファイル）の変更のみ。
それ以外の状態変更（ファイル編集・作成・削除・設定変更・外部送信等）は実行してよく、出力契約で事後報告する。
```

## 7. Cursor 対話レーンでの適用（相対マッピングのみ）

同じ委譲ラダーを Cursor でも使う。固有のバージョン名はここにも書かない（§1どおり）。

| 相対階層 | Cursor での読み替え |
|---|---|
| 最上位（指揮者） | その時点の対話で選んだ最上位モデル |
| 軽量〜中位（ワーカー） | 高速・低コスト寄りのモデル |
| 差配手段 | 親が Task / サブエージェントで委譲（ワーカーが上位を横付け召喚しない） |

- **指揮者セッション**: §2どおり着手前に委譲を設計し、物量はワーカーへ、統合・検証は本人
- **非最上位セッション**: §2の委譲が必要なら、着手前にユーザーへ最上位への切替を1行で提案する
  （製品にモデル自動切替APIはない。ブロックや専用Skillで強制しない）
- 昇格提案の目安は §2 と同じ: 3ファイル以上／機械的反復、設計判断・横断検証、行き詰まり
- Cursor 実効ルールは `.cursor/rules/` に薄く置く。本書の全文は複製しない
