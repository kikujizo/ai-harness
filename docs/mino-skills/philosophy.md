# mino-skillsの設計仮説

この文書は、ミノ駆動氏の公開資料から何を読み取り、どのような効果を期待して6つのSkillへ再構築したかを説明します。
公開資料に直接書かれている内容と、本リポジトリでの再構築上の判断を分けて記載します。

> mino-skillsは公開情報からの推定再構築であり、本人の非公開Skill本文の複製や公式実装ではありません。

## 結論

再構築の中心仮説は、AIへ一つの巨大な設計プロンプトを与えるより、目的の異なる思考工程をSkillとして分け、各工程に入力・出力・反証条件を持たせる方が判断の混線を減らせる、というものです。

`mino-context-discovery`では出力生成と追跡可能性を確認しましたが、境界候補の正確性や人間の判断への有用性は未確認です。PR #42/#43ではSkill定義の欠陥を外部要件レビューが検出しており、有用性の証拠にはできません。6 Skill全体の有効性や他案件での再現性も未確認です。

## 6 Skillへ分けた理由

| 工程 | 分ける理由 | 期待する効果 |
|---|---|---|
| 要求定義 | 人間の目的を決める工程と、解決策を作る工程を混ぜない | 手段を目的と誤認するのを防ぐ |
| 境界発見 | 用語の意味の衝突を、業務時系列とは別に扱う | 同じ言葉で別物を扱うモデル混在を見つける |
| イベント分析 | 起きたこと、トリガー、例外を時間順に扱う | フローと集約候補の関係を見える化する |
| モデル深化 | 実装中ではなく設計Checkpointで前提を疑う | 思いつきの再設計を避けながら代替案を比較する |
| 契約駆動実装 | 上位仕様から条件を導出してからコード化する | AIがIssueにない仕様を補うのを防ぐ |
| 変更容易性レビュー | merge判定とは別の補助レンズにする | 設計上の気づきを標準レビューへ渡す |

## 現時点の限界

- `mino-context-discovery`以外は、GitHub上で追跡できる実行例が不足している。
- `mino-context-discovery`も、出力の正確性、人間の判断への有用性、通常プロンプトとの差は評価していない。
- PR #42/#43では外部要件レビューがSkill定義の欠陥を検出した。これは被検査対象の失敗記録であり、Skillの有用性を示さない。
- PR #33には6 Skillを「検証済み」とする記述があったが、根拠の多くは移送対象外の記録に依存しており、本Issueでは観測済みと扱わない。
- 6 Skillを順番に使うと有効か、単体利用が有効かは比較できていない。
- モデルやAI環境が変わった場合の再現性、所要時間、偽陽性率は測定していない。
- 公開資料の考え方から本リポジトリの具体手順へ変換した部分は、本実験固有の仮説である。

これらの限界を埋めるまでは、6 Skillをlab・明示呼び出し限定で運用します。
