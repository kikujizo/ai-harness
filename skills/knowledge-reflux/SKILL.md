# knowledge-reflux: 知見を実績ベースで昇格させる

目的関数は「同じ知見を二度と会話で説明しない」。
このスキルは**提案までで止まる（恒久L1）**。常時適用ルールの書き換えは人間の承認が源泉。
昇格パイプラインと数値ゲートの正本は `docs/harness/knowledge/reflux.md`。

## 手順

### ステップ1: 今回のセッションで得た知見を列挙する

セッション・レビューで「再説明した」「つまずいた」「基準外だと感じた」点を箇条書きにする。
それぞれに「これは何回目の参照か」を付ける（体感でよい）。

### ステップ2: 昇格ゲートを判定する（`docs/harness/knowledge/reflux.md`）

各知見を数値ゲートに当てる。

- **ゲート①・同じ知見を2回参照した** → `knowledge/criteria/`（または `docs/criteria/`）への **新規criteria化を提案**。
  バイナリ（○/×） + 観測手順に開けるもののみ。開けない知見は `docs/` に読み物として留める。
- **ゲート②・レビューで「基準外の気づき」が3回同内容** → **既存の基準ファイルへの項目追記を提案**。
- **ゲート③・基準がプロジェクト・ツール横断で3回以上×を検出した実績** → **AGENTS.md への追記を提案**。
  プロジェクト固有の基準は昇格させない（そのリポジトリの `docs/criteria/` 止まり）。
- **1回だけの着想** → まだ昇格しない。生メモ／`docs/` に置く。
- ゲート未達のものは提案しない（先回りして基準を量産しない）。

### ステップ3: 提案のみ行い、人間の承認を待つ（L1固定）

昇格候補を「知見 → 提案する昇格先 → 根拠（何回目の参照・×の実績）」の形で人間に提示する。
自分で criteria・AGENTS.md を書き換えない。承認されるまで待つ。

### ステップ4: 承認されたら反映し、旧記述を削除する（経緯レス正本）

承認された昇格だけを `knowledge/`（または該当リポジトリ）に反映する。反映時に:

- 上位に昇格させたら、**置き換えた旧記述はその場で削除**する（アーカイブへ移す）。
- 上位ファイルに「旧版・移行済み・参照しない」の注記を残さない。
- 経緯は Decision Log（`docs/templates.md`）に1行残す。正本には「今の正解」だけを置く（詳細 `docs/harness/knowledge/ssot.md` §経緯レス正本ルール）。

## トークンガード

- 昇格ゲート未達の知見は提案自体をしない（人間の注意力とトークンの無駄）。
- 剪定（6ヶ月×なし・1年参照なし）に気づいたら、削除候補として合わせて提案する。削除も Decision Log 必須。
