# mino-context-discovery — 言語ゲームによる境界発見

明示指定されたとき、ドメイン内で同じ言葉が文脈により違う意味・違うルールを持つ「意味の断層」を特定し、断層線に沿って境界付けられたコンテキストを引き、コンテキストごとのユビキタス言語を定義する。

**コア原則**: 全文複製禁止。規範的正本は [docs/mino-skills/core/mino-core-principles.md](../../../docs/mino-skills/core/mino-core-principles.md) を参照する。

## 使いどき / 使わないとき

**使う**（明示指定時）: 用語・意味・境界の衝突が主問題のとき。新規ドメイン分析／「同じ用語なのに話が噛み合わない」／モデル混在の予防・診断。

**使わない**: 業務時系列・例外フローが主問題（→ `mino-event-storming`）／単発の問い直し（→ `reframe-question`）／複数ターン要求定義（→ `mino-socratic-requirements`）／Issue #38 対象の下流Skill。

## `reframe-question` との境界

単発の問い直しは `reframe-question` を優先。本 Skill は用語×文脈の意味マトリクスと境界付けが主成果物。


## 四つ組の定義（記入基準）

| 要素 | 一言定義 | 記入例 |
|---|---|---|
| アクター | その言葉を使う人・役割・外部システム | 在庫スタッフ |
| 目的 | そのアクターが達成したいこと（「なぜ？」で遡った上位目的） | 商品在庫の調整や棚卸し |
| 文脈 | どの状況・時点・場面での使用か。同じアクターでも場面が変われば意味は変わりうる | 入荷〜棚卸し〜出荷の作業中 |
| ルール | その意味であるとき、何をして良く・何をしてはいけないか | 入出庫・在庫調整のルールに従う |

## インプット

必要なもの: ドメインが分かる素材（仕様書・議事録・業務説明・ヒアリングメモ・既存ドキュメント。断片的でよい）。

足りないとき: 推測で埋めず、次の順で人間に聞く（**一度に3問まで**）: ①関わる人・役割・外部システム（アクター） ②それぞれが達成したいこと（目的） ③「話が噛み合わなかった」「認識違いでトラブルになった」場面の実例。

素材が分析に足りない場合は強行しない。表を無理に埋めず、「素材不足」を第一所見として、必要な素材・ヒアリング項目を確認事項に列挙して終える（分量の多い出力は正しさの証拠ではない）。

対話できる環境では、✅承認ゲートで出力を止めて人間の返答を待つ（返答なしに次のSTEPへ進まない）。対話できない実行環境（バッチ実行・非対話エージェント）では、仮定したことを出力の「保留事項」に明記して続行する。承認ゲートは「確認待ち」ではなく「確認事項の列挙」として出力する。

## 手順

### STEP 1: アクターと目的の棚卸し

- 素材からアクター（人・役割・外部システム）を列挙する。
- 各アクターの目的を1行で言語化する。目的が手段に見えたら「なぜ？」で上位目的へ遡る（例: 「バーコードを読む」→なぜ→「棚卸しで在庫数を正確に保つ」）。
- 同一人物でも役割が違えば別アクターに分解する（兼務は分ける）。

**✅承認ゲートG1**: アクター×目的の一覧を提示し、抜けがないか確認する。特に見落としやすいのは「発言力の弱いアクター（現場作業者・CS）」「外部システム・外部業者」「時間差で現れるアクター（監査・経理・返品対応）」。

### STEP 2: 用語×文脈の意味マトリクス（言語ゲーム分析）

- 素材から中核用語を10〜20個抽出する。優先するのは「頻出する語」「揉めたことのある語」「意味の広すぎるクソデカ語（例: 商品・ユーザー・完了・対応）」。
- 用語×アクター文脈の表を作り、各セルに「その文脈での意味」と「その意味に付随するルール」を書く。
- セルを埋めるときの問い: **このアクターの目的にとって、この言葉は何を指すか？ その意味であるとき、何をして良く、何をしてはいけないか？**
- 意味またはルールが文脈間で変わるセルを「断層」としてマークする（⚡等）。
- 素材が薄く根拠のある語が10個に満たない場合は水増しせず、採用件数とその理由を明記する。網羅より断層の実証を優先する。
- マトリクスの非該当セル（その文脈で中核語でない・言及がない）は「—」で省略してよい。
- この表は境界発見のためのワーク成果物（アクター文脈単位）。最終出力では境界確定後にコンテキスト単位へ再編して提出する（出力契約2）。

### STEP 3: 断層線に境界を引く

- 断層が集中する線に沿ってコンテキスト候補を切る。境界の単位は「大粒度の目的」——目的が変わる所で言葉の意味とルールが変わる。
- 各候補に目的が伝わる名前を付ける（「〜管理」「共通」のような無目的名を避ける）。

**✅承認ゲートG2**: 境界仮説を、根拠となる断層（どの用語の意味がどう変わるか）とセットで提示し、合意を取る。

### STEP 4: コンテキスト定義（四つ組＋ユビキタス言語）

- 各コンテキストを四つ組（アクター／目的／文脈／ルール）で記述する。
- コンテキスト内ユビキタス言語表を作る: 用語／このコンテキストでの意味／付随ルール／他コンテキストでの対応語（翻訳）。
- 1つの用語が同一コンテキスト内で2つの意味を持ったままなら、境界かモデルがまだ粗い。STEP 3へ戻る。

### STEP 5: コンテキスト間関係

- コンテキストマップを描く: どの用語が境界を越えるか、越えるとき何に翻訳されるか、依存の向き（上流/下流）。

### STEP 6: 反証ラウンド（必須・スキップ禁止）

以下すべてに明示的に答える。1つでも該当したらSTEP 2〜3へ戻って修正する。検討の結果「修正せず維持」と判断した場合も、該当の事実と維持の理由を「判断根拠」に記録する（無言スルーのみ禁止）。

1. この境界は**組織図**をなぞっただけではないか？（部署≠目的）
2. **DBテーブルや既存システム構成**をなぞっただけではないか？
3. **画面・UIの単位**をなぞっただけではないか？
4. 同じ語が同一コンテキスト内で二重の意味を持っていないか？（同音異義の見逃し）
5. 別の語が同じ概念を指していないか？（異音同義の統合漏れ）
6. アクター1人＝1コンテキスト（部署と1:1）になっていないか？ 判定テスト: (a) 隣り合う2つの目的を同一人物が同時に果たせるか (b) その境界に「意味またはルールが変わる用語」の実証があるか。実証があれば部署と1:1でも維持してよい（理由を記録）。実証がなければ統合を検討する。
7. **時間・状態の変化**で意味が変わる用語を見逃していないか？（例: 「注文」は出荷前と出荷後で従うルールが別物）
8. 「共通」「管理」「マスタ」という名前のコンテキストを作っていないか？（目的の言語化から逃げた兆候）

## 出力契約（この形式以外で出力しない）

1. **コンテキスト一覧表**: `| コンテキスト | アクター | 目的 | 文脈 | 主要ルール |`
2. **用語×コンテキスト意味マトリクス**（断層マーク付き）＋**コンテキスト別ユビキタス言語表**（STEP 4の成果物。境界確定後のコンテキスト単位に再編したもの）
3. **コンテキストマップ**（Mermaid `graph TD`。エッジに「越境する用語→翻訳」を記す）
4. **判断根拠・確認事項・保留事項**: 各境界がどの素材・発言から導かれたか／**確認事項**（人間への質問。非対話環境の承認ゲートG1・G2はここに列挙する）／**保留事項**（実行者が置いた仮定）——確認事項と保留事項は性質が違うため区別して書く／反証ラウンドの該当・修正・維持の記録
5. **ネクストアクション**: 時系列分析すべき業務候補（→ `mino-event-storming`）の指名。深化させるべきモデル候補は、現時点では未導入のIssue #38対象Skill（`mino-model-deepening`）へ自動連鎖しない。Issue #38完了後、かつ `.agents/skills/mino-model-deepening/SKILL.md` が実在する場合のみ、条件付き候補として明記する。導入前は「後続Issue候補」または通常の設計工程として出力する。

## アンチパターンガード

- **技術層で切らない**: フロント/バック/DBは手段の区分。境界は目的で切る。
- **全社統一辞書を作らない**: 「同じユビキタス言語が適用可能な範囲は、境界付けられたコンテキストまで」。全文脈共通の唯一の定義を強制した時点で言語ゲームの否定になる。
- **境界＝マイクロサービスと即断しない**: 論理境界の発見と物理分割は別の意思決定。
- **数の異常を疑う**: コンテキストがアクター数より極端に多い／ドメイン全体で1つしかない場合は、切り方の粒度（目的の大きさ）を再検討する。ただし再検討の結果「このドメインは単一コンテキストが正しい」もまた正当な結論であり、分割のノルマはない。
- **AIだけで確定しない**: 問題定義フェーズは人間pilot。承認ゲートを飛ばして「完成品」として納品しない。

## 品質基準（Doneの定義）

- 各コンテキストが四つ組で1画面以内に説明できる。
- 境界ごとに「意味またはルールが変わる用語」が最低1つ実証されている（なければその境界は根拠不足として要再検討）。
- 出力契約の5点が揃い、すべての境界の根拠が素材の記述に遡れる。
