# mino-skills 検証サマリー

この文書は、公開情報から再構築したmino Skillについて、GitHub上で追跡できる試行とレビューから、確認できた事実と未確認の有用性を分けて記録します。
結論は`観測済み／仮説／未確認`に分け、根拠がない成功を補いません。

## 結論

- `mino-context-discovery`は、1件の出力生成とGitHub上の追跡可能性を**観測済み**です。
- `mino-context-discovery`の境界候補の正確性、人間の判断への有用性、通常プロンプトとの差、別案件での再現性は**未確認**です。
- `mino-socratic-requirements`は、Issue #46で10問の対話出力を生成し、人間が承認する問題文へ到達したことを**観測済み**です。
- `mino-socratic-requirements`が通常の対話型仕様化より優れるか、別案件でも再現するかは**未確認**です。今回の人間評価では、最大10問を1問ずつ、人間回答と確認往復で積み上げる現行方式は再利用しないと判断しました。
- その他4 Skillの出力生成、正確性、有用性は**未確認**です。
- PR #42/#43はmino Skill自身の欠陥をChatGPT要件レビューが検出した記録であり、有用性の証拠には含めません。

したがって、現時点では6 Skillすべてについて一般的な有用性は未確認です。`mino-socratic-requirements`はlabのまま凍結し、通常業務では提案・実行しません。他のSkillもcore昇格や一括利用を推奨しません。

## 判定区分

| 区分 | 判定条件 |
|---|---|
| **観測済み** | GitHub上の入力、出力、レビュー、修正へ遡れる |
| **仮説** | 公開資料または観測結果から期待できるが、実行証拠がない |
| **未確認** | 判断に必要な入力、出力、比較対象が不足している |

## 既知の限界

- `mino-context-discovery`は単一の作例で、非対話実行である。
- 人間による境界合意、通常プロンプトとの比較、別案件での再現確認がない。
- 利用モデル、所要時間、出力量、実務上の負担を比較していない。
- `mino-socratic-requirements`は単一事例で、通常条件は追加質問なしのため、Skill固有の質問選択効果を分離できていない。
- Issue #46のソクラテス条件は人間概算で約15分だが、開始時刻を記録しておらず、baselineの1分20秒との厳密な所要時間比較はできない。
- Issue #46では背景説明なしに質問を開始して一度再開始しており、運用ラッパーの影響を含む。
- PR #33の検証済み表記は、GitHubから確認できない記録に依存するため採用していない。

## 実例から得た判断

### 残す

- **証拠区分**: 出力生成、正確性、有用性を別の主張として判定する。
- **lab運用**: 有用性が未確認のSkillは明示呼び出し限定とする。ただし、凍結したsocraticは下記の凍結後運用を優先し、通常業務で実行しない。
- **外部レビュー**: PR #42/#43でSkill定義の欠陥が見つかったため、mino Skill自身を検査手段とみなさず独立レビューを続ける。
- **socraticの判断構造**: AIが問題候補を提示し、人間が選ぶゲートと反証による補正は、Issue #46の承認済み問題文に寄与した観測として残す。

### 修正する

- 検証済みという表現は、GitHub上の入力・出力へ遡れる場合だけ使う。
- 追跡可能性と有用性を同じ判定にしない。
- PR #42/#43を有用性の証拠から除外し、既知の欠陥として扱う。
- 1件の成功例からSkill全体や他案件での再現性を断定しない。
- 各実行記録に、入力、期待、観測、失敗、保留、利用モデルを可能な範囲で残す。
- 現行Skill本体は変更しない。実案件で類似ニーズが発生した時だけ、ChatGPTが既知情報から推論案を先に出し、少数・選択式で確認する軽量版を提案する。

### 見送る

- 6 Skillすべてを固定順で自動実行する運用。
- PR #33の「全Skill検証済み」という状態表記の無条件な移送。
- 実行証拠がない段階でのcore昇格や標準フローへの組み込み。
- Issue #46の単一事例だけを根拠にしたsocraticのcore昇格・自動発動。
- 最大10問を1問ずつ、人間回答と確認往復で積み上げる現行`mino-socratic-requirements`の通常業務での提案・実行。
- 現行Skillの能動的な改修Issue・追加実験の起票。

## 次の検証候補

1. **Skill選択**: 同じ業務素材にcontextとeventを別々に適用し、境界問題と時系列問題を混同せず分けられるか確認する。
2. **設計から契約**: 小さな既存モデルをmodel-deepeningで見直し、人間が採用した案だけをcontract-driven-codingの契約表へ渡す。
3. **変更容易性レビュー**: 既知の設計欠陥を埋めた小さなコードで、6観点の検出数、見逃し、偽陽性（問題でない箇所を問題と判定すること）を記録する。

`mino-socratic-requirements`は次の検証候補から外す。実案件で類似ニーズが発生し、ChatGPTが提示した軽量案を人間が採用した場合に限り、別Issueで再設計する。

## mino-socratic-requirementsの凍結後運用

- labのまま凍結し、通常業務では現行Skillを提案・実行しない。
- Skill本体は今回変更しない。
- 能動的な改修Issue・追加実験は作らない。
- 実案件で類似の要求整理ニーズが発生した時だけ、ChatGPTがAI推論・少数質問・選択式の軽量案を提案する。
- 人間が軽量案を採用した場合に限り、別Issueで再設計する。

各検証は1 Skillまたは1連携に限定し、結果が悪い場合も削除せず記録します。
