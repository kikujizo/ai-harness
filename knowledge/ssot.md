# 正本マップと鮮度ガード — SSOT Rot を防ぐ

情報が複数の場所に散らばると、AIはどれが今の正解か分からず「古いまなざしを内面化した精密な間違い」を量産する。
これを **SSOT Rot（正本の腐敗）** と呼ぶ。Context Rot（文脈の腐敗）が長い会話で起きるのに対し、
SSOT Rot は正本管理の不備で起きる。モデルの性能でも基準の質でもなく、**正本マップと鮮度確認の運用**だけが防げる。

## 1. 正本マップの方法論

「どれを読めば今の正解か」を **1枚の表** で答える地図を持つ。原則は3つ:

- **情報の所在を1枚にまとめる** — 情報の種類ごとに「正本の場所」を1行で書く。
- **矛盾時は表が勝つ** — 実物と表がずれていたら、まず表を疑い、表を直してから作業する。読み手は迷ったらこの表に従う。
- **表を肥大させない** — 世代が変わる出来事（設計の改版・フォルダ統合・大きな移行の完了）が起きたら、作業前にこの表を更新する。行が増えて更新が面倒になったら行を減らす（**読まれない正本マップは腐る**）。

### 正本マップ テンプレート（空欄）

| 情報の種類 | 状態 | 正本の場所 |
|---|---|---|
| 設計思想・仕様 |  |  |
| 常時適用ルール（AGENTS.md 等） |  |  |
| テンプレ・プロンプト |  |  |
| 基準ファイル（criteria） |  |  |
| 実装の現在地（何が動いているか） |  |  |
| 進行中の作業・移行の進捗 |  |  |
| アーカイブ（参照禁止・差分確認時のみ） |  |  |

正本マップ自体は「地図の地図」なので、腐ると被害が最大になる。月1回、または構造変更時に、この表と実態の照合だけを行う（中身の再レビューはしない）。

## 2. 鮮度ガード — 長い推論・大きな成果物を作る前の3問

外部情報を用いた再帰的推論は、外部信号が古いと**古い前提で精密な間違い**を量産する。
長時間の推論・大きな成果物作成の**前に**、この3問だけ確認する:

1. **手元は最新に同期されているか？** — 最新の状態を取得済みか（例: 最新 main を pull したか、共有ノートが最新か）。不安があれば人間に一度だけ確認してよい。
2. **正本マップと矛盾しないか？** — これから読む資料が正本マップの「正本の場所」列と一致するか。参照禁止（アーカイブ）行を根拠にしていないか。
3. **実機と文書のどちらが正か？** — 文書（設計）と実機（リポジトリ・実際の設定）が食い違ったら、**事実の記述は実機が正、意図の記述は文書が正**。実機を読める場合は読んでから書く。

## 3. 経緯レス正本ルール

**正本の場所に「参照しない・旧版・移行済み」の注記や旧版ファイルを置かない。**

- 差し替えたら、旧記述・旧ファイルは**その場でアーカイブへ移す**（正本に残さない）。
- 移行期にどうしても残す時限注記には、**期限とクリーンアップ手順を必ず併記**する。
- 経緯の記録はアーカイブと Decision Log が担い、正本にはAIが実運用で読む「今の正解」だけを置く。ツギハギ注記の放置は、AIの無駄な読み込みと世代誤認の温床になる。

## 4. 事故ったときの回復手順

同期ずれ・世代誤認に気づいたら:

1. 作業を止め、何を前提に何を作ったかを列挙する。
2. 誤った前提で作られた成果物は「修正」より先に**仕分け**する（全面改訂 / 部分修正 / 破棄）。誤り前提の成果物を惜しんで直すと、直し漏れが次の世代誤認の種になる。
3. 誤認の原因を「正本マップの不備」か「鮮度ガードの不備」に分類し、該当側を1行更新する。
4. 破棄・改訂の判断を Decision Log に残す（同じ誤認の再発時に前例が効く）。
