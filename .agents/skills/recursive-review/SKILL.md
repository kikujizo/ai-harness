# recursive-review: 基準を先に、照合を名指しで

チェックリストを「参照する」のではなく、レビュー直前に基準を復唱して「見る目」にしてから対象を見る。

**前提（独立性チェック）**: レビュー対象が自分（同一AI）の実装物なら主レビュアーを引き受けない
（正本: ルート`AGENTS.md`「レビュー独立」）。観点整理・第二意見依頼素材の作成に切り替え、verdictは独立レビュアーに委ねる。

## 手順

1. **基準の復唱**（レビュー対象を開く前に行う）
   - PRレビュー: 元Issueの受け入れ条件を箇条書きで復唱
   - コード（diff/PR）レビュー: 上記に加えて `docs/criteria/code-review-criteria.md`（第1層・バイナリ基準）を復唱し、
     `docs/criteria/quality-lens.md`（第2層・走査レンズ）を通読時の「見る目」に使う
   - 文書レビュー: `docs/criteria/` の該当基準ファイルを読み、要点を3行で復唱
   - セキュリティ確認: AGENTS.mdの安全ルール+不可逆4カテゴリ（高リスク変更リスト）を復唱
   - 基準が存在しない場合: 対象分野の暫定基準を10項目以内で作り、当該レビューにそのまま使ってよいが保存はしない。
     有用だったと分かったら knowledge-reflux のゲート①（同じ知見を2回参照）を経て正式化する
2. **照合**: 基準1項目ずつ対象に当て、○/×/確認不能を付ける。
   ×は「基準番号 + ファイル:行（文書なら該当箇所の引用）」で名指しする。
   引用・証拠は**帰属**（誰の主張か・どの資料のどちら側の記述か）まで確認する。
   帰属の反転を見逃すと結論ごと逆転する（実例: ai-dev-workflow PR #45 でmergeブロック級の帰属反転を検出）。
3. **横断視点**: 基準外で気づいた問題は「基準外の気づき」として分離して書く
   （基準そのものの改善候補になる）。
4. **判定と次アクション（ディスポジション契約）**: merge可能/修正必須/保留。**全指摘**（×・確認不能・基準外の気づき）に
   必ず三択を割り当てる: **今回修正**（スコープ内で軽微なものの既定。AI実装者の修正コストは低く後回しの経済的理由がない）/
   **wontfix**（理由1行を記録）/ **後回し**（**追跡Issue URL必須** — URLを添えられない後回しは禁止。追跡されない後回しは消滅する）。
   未割り当ての指摘が残る限り `approve` を出さない（「軽微な指摘はあるがマージ可」の禁止）。
   GitHub書き込み権が無い文脈（読み取り専用・委任サブエージェント）では、後回し指摘を
   書き込み可能な独立AIへ再ルーティングする。候補がなければ `blocked` を記録する
   （人間への作業依頼ではなく、AI PMの判断キューへ返す）。
5. **再照合（最大1回）+ 無進展検出**: 修正が返ってきたら×だった項目だけ再照合する。全体を見直さない。
   **前回と同じ逸脱リストになったら即停止**し「このループでは直せない構造的問題」として
   AI PMへ返す（独立AIへの再ルーティングまたは `blocked`）。
   「前回の逸脱リスト」は前回レビュー出力（GitHubコメント/引き継ぎ資料）から復元する。
   復元できない場合は、再照合で×項目数が減っていないことをもって無進展とみなす。

## 出力形式

`docs/templates.md` の「レビュー結果」フォーマットに従う。
最終行に `REVIEW_VERDICT: {approve|request-changes} [risk=high]` を1行で付ける（AI間機械伝達用）。
必要時は REVIEW_VERDICT 直前に補助行（SUBJECT_VERDICT / ARTIFACT_READINESS）を置ける。
**形式・補助行の定義の正本はルートの `AGENTS.md` のverdict節**を参照すること。
最後に非エンジニア向けの1文（このPR/文書は何で、何が問題か）を付ける。

## トークンガード

- このスキルは品質が重要な対象だけに使う。軽微なtypo修正PRには使わない
- 基準ファイルは再利用する。毎回ゼロから基準を作らない
- 読むのは対象diff/文書と基準だけ。関連ファイルの芋づる読みをしない
- 照合をサブエージェントへ委任する場合は、復唱した基準を委任プロンプトへそのまま埋め込む
  （下位モデルはSkillを自発発動しない — 2026-07-17実測 0/3）
- 「基準外の気づき」が3回続けて同じ内容なら、基準ファイルへの追記をAI PMが変更Issueとして提案する
  （レビューのたびに同じ指摘を書くのはトークンの無駄）。この判定は `docs/harness/knowledge/reflux.md` のゲート②にあたる
