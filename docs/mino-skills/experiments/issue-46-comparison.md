# Issue #46 通常仕様化とソクラテス問答の初回比較

同じ固定要望に対する通常仕様化と`mino-socratic-requirements`を比較した結果、ソクラテス問答はこの1件では人間が承認できる具体的な問題を特定しました。一方、約15分を要した現行方式は実務利用に値しないと人間が判断したため、Skillをlabのまま凍結します。

## 比較条件

| 項目 | 通常仕様化 | ソクラテス問答 |

| 実行 | Codexによる一括仕様化、追加質問なし | CodexでSkillを明示実行、1回1問 |

| baselineの開示 | ソクラテス問答完了まで人間・問答実行者へ非開示 | 完了後に比較担当だけが開示 |
| 証拠 | [baseline](issue-46-baseline.md) | [問答ログと要求定義書](issue-46-socratic.md) |
| 最終記録SHA-256 | `62908EEDE0912292CEFFA72B60952B5EB9A2216FC4942AFBA64928CCA10347E0` | `71EE39F1692D0A4C08C46A740B4551EA21E832F37DB3EB6BAAD62B24D9A7100E` |

通常仕様化は1分20秒で完了しました。ソクラテス問答は、AIの応答待ち時間を含む人間の概算で約15分を要しました。概算では約11倍ですが、ソクラテス条件の開始時刻を機械記録していないため厳密な実測値ではなく、差のすべてがSkill固有の要因とは断定しません。

## 追加情報量と質問選択の分離

### 人間の追加情報による効果

Codexの5時間枠、平均トークン消費、レビュー1件の境界、見落とし頻度、未計測の現状、許容できる記録負担、PR優先、ラリー回数は、人間の回答で初めて得られました。ソクラテス側が詳しい主因の一つは、通常仕様化には与えなかった追加情報です。

### 質問選択について観測できたこと

10問すべてが、承認済み問題文、成功基準、未確認前提、または次段問題のいずれかに使われました。特に対象を絞る質問1・9、主指標を選ぶ質問2、レビュー単位を決める質問3、基準線の欠如を確認する質問7は、最終問題文へ直接つながっています。

これは、この1件で質問が関連情報を引き出した証拠です。ただし、同じ質問数を使う通常の対話型仕様化との比較がないため、質問選択がSkill固有に優れているという結論は**未確認**です。

## 機能した点

- 1回1問により、対象、指標、品質条件、測定対象を順に確定できた。
- 問題候補を複数提示し、人間が主問題と次段問題を分離できた。
- 反証ラウンドにより、0件という目標と、頻度増加を不合格とする基準を分離できた。
- baselineが先に示した解決案へ引っ張られず、計測基準線の欠如を主問題に選べた。

## 機能しなかった点・負担

- 背景と実験目的を示さず質問を開始したため、人間が質問意図を理解できず一度再開始した。
- 各回答の要約確認は誤解修正に役立った一方、10問を超える往復を必要とした。
- 質問6は当初「何件まで」という数値回答を求め、人間の補足を経て前後頻度の比較へ修正された。
- 人間概算で約15分を要した。開始時刻を記録していないため、baselineとの厳密な所要時間比較はできない。

## 判断

- **残す**: Skillはlabのまま凍結し、問題候補を提示して人間が選ぶゲートと反証による補正を実験記録として残す。
- **修正**: 今回はSkill本体を変更しない。実案件で類似ニーズが発生した時だけ、ChatGPTがAI推論・少数質問・選択式の軽量案を提案する。
- **追加検証**: 能動的には行わない。人間が軽量案を採用した場合に限り、別Issueで再設計と検証を判断する。
- **見送る**: 最大10問を1問ずつ、人間回答と確認往復で積み上げる現行方式の通常業務での提案・実行、core昇格、自動発動、能動的な改修Issueの起票。

## 人間評価

比較後、人間は「C：再利用したくない」と回答した。理由は、最大10問を1問ずつ、人間回答と確認往復で積み上げる現行方式の負担である。一方、少数の質問でAIが推論・提案し、人間が選択肢から答えて進める方式なら利用余地があると補足した。

現行Skillはlabのまま凍結し、通常業務では提案・実行しない。実案件で類似ニーズが発生した時だけChatGPTが軽量案を提案し、人間が採用した場合に限り別Issueで再設計する。
