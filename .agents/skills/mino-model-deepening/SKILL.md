# mino-model-deepening — 脱構築とスキーマ理論によるモデル深化

既存のドメインモデルを支配している暗黙の前提（スキーマ）を疑い、「みんなは○○と呼んでいるが正体は××ではないか？」と脱構築を繰り返して、浅いモデルを深いモデルへ進化させる。フラグ・種別コード・例外処理の増殖は、既存スキーマが現実に合わなくなっている兆候として扱う。

理論的裏付けは「脱構築」と呼ばれる概念の問い直しと、認知科学のスキーマ理論。なお本スキルの「脱構築」はミノ駆動氏のX上の用法に倣った通俗的な意味（固定化した呼称を疑い、概念を捉え直すリフレーミング）であり、デリダの哲学用語の厳密な意味ではない。ミノ駆動氏の公開発信にインスパイアされた非公認の再構築（出典は本スキル群のcore/mino-core-principles.md）。

## 使いどき / 使わないとき

**使う**: モデルにフラグ・種別コード・例外処理が増殖してきたとき／「なんとなく気持ち悪いモデル」を言語化したいとき／event-stormingで出た集約候補を磨くとき／同名別物・別名同物の疑いがあるとき。

**使わない**: 境界・用語の発見（→ mino-context-discovery）／業務フローの洗い出し（→ mino-event-storming）／コード実装（→ mino-contract-driven-coding）／ゼロからのモデリング（既存モデルがない場合はまず mino-event-storming から）。

## コア原則（mino-*共通）

1. **言語ゲーム**: 言葉の意味は定義ではなく、文脈と目的の中での使われ方で決まる。定義論争から入らず「誰が・何の目的で・どう使うか」から入る。
2. **四つ組**: 意味理解の最小確認単位は「アクター／目的／文脈／ルール」。この4つが揃わない用語理解は未完成として扱う。
3. **目的駆動**: 構造は目的に従う。境界付けられたコンテキストとは「大粒度の目的」である。迷ったら「なぜ？」で上位目的へ遡る。
4. **スキーマ理論**: 人もAIも既存の認知枠組みで対象を見る。既知の概念への執着がモデルの成長を止める（「悪魔」と呼び続ける限り病原菌は発見できない）。
5. **脱構築**: 「みんなは○○と呼んでいるが、正体は××ではないか？」と問い続ける。固定化した言葉に構造を引っ張らせない。
6. **均質性制御**: AIの出力は学習内容の平均＝均質に収斂する。だから区分（スキル）を切り、区分内では出力契約で均質化させ、区分間の濃淡・軽重の設計は人間が握る。
7. **人間pilot原則**: 問題定義フェーズは人間がpilot、AIはcopilot。AIは問い・選択肢・根拠を出し、決定はアクター（人間）に委ねる。
8. **根拠の遡行可能性**: すべての結論に「どの素材・発言から導いたか」を添える。遡れない結論は仮説として明示する。

## インプット

必要なもの: 既存のドメインモデルが分かる素材（概念定義書・ER図・クラス図・API仕様・企画仕様書の用語表・event-stormingの集約候補メモ。断片的でよいが「概念・属性・関係・ルール」のいずれかが読み取れること）。

足りないとき: 推測で埋めず、次の順で人間に聞く（**一度に3問まで**）: ①対象モデルの一次資料のありか（仕様書・ER図・コード） ②最近「気持ち悪い」「なんか違う」と感じた具体的な箇所や事象 ③既存モデルを作った経緯（何を参考にした・何を踏襲したか）。

対話できる環境では、✅承認ゲートで出力を止めて人間の返答を待つ（返答なしに次のSTEPへ進まない）。対話できない実行環境（バッチ実行・非対話エージェント）では、仮定したことを出力の「保留事項」に明記して続行する。承認ゲートは「確認待ち」ではなく「確認事項の列挙」として出力し、採否判断はせず複数案を採点付きで併記する。反証ラウンドで「STEPに戻る」場合は、修正済みの最終形のみを提出し、戻った過程は判断根拠に記録する。

## 手順

### STEP 1: 現行モデルの棚卸し

- 概念・属性・関係・ルールを列挙する。
- 各概念について「この概念がうまく説明できていること」と「説明できずにフラグ・種別コード・例外処理・注記で逃げていること」を対で記録する。後者が深化の入口。
- 棚卸しの粒度は素材依存でよいが、中心概念（他の概念から参照される・命名の議論が多い・変更頻度が高い、のいずれか1つでも満たす概念＝OR条件）には印を付け、STEP3で優先的に脱構築の対象にする。

### STEP 2: スキーマ検出（悪魔候補の特定）

- このモデルを支配している暗黙の前提・借り物のメタファーを列挙する（「このモデルは何のアナロジーで作られたか？」紙帳票の踏襲・Excelの写し・他社サービスの模倣・昔のシステムの移植など）。
- 増殖するフラグ・「その他」区分・肥大する注記は、既存スキーマが現実に合わなくなっている兆候（＝病原菌を「悪魔」と呼び続けている状態）としてマークする。
- 「なぜこの前提を置いたか」を遡れるか確認する。遡れない前提（「昔からそうだから」「なんとなく」）ほど脱構築の優先度が高い。

### STEP 3: 脱構築ラウンド

- 中心概念それぞれに対し「みんなは○○と呼んでいるが、正体は××ではないか？」の代替案を最低3つ強制生成する。
- 生成の観点:
  - (a) 時間で分ける（「注文」は出荷前と後で実は別概念では？）
  - (b) 目的で分ける（同じ「アイテム」でも演出目的と経済資産では別では？）
  - (c) 統合する（別々に見える2概念は同一概念の異なる現れでは？）
  - (d) 出来事として捉え直す（「状態」ではなく「起きたこと」の履歴では？）
  - (e) 役割として捉え直す（「人」の属性ではなく「役割」では？）
- 代替案は最低3つ。5観点すべてを無理に使わなくてよい（素材の裏付けが薄い観点は「該当薄」と明記してスキップ可。根拠なき案出しは水増し）。

### STEP 4: 評価

各代替案を4基準で採点する。

| 基準 | 問い |
|---|---|
| ①説明力 | STEP1で見つけた例外・フラグがいくつ消えるか |
| ②不変条件の置き場所 | ルールの責務がどの概念に属するか明確になるか |
| ③言語ゲーム整合 | 現場の言葉と接続できるか（誰も使わない完全な造語は減点） |
| ④移行コスト | 既存データ・既存コードからの移行がどれだけ現実的か |

採点は各基準1〜5点。合計点は参考値とし、推奨の決定は定性判断を優先する（点数の機械的採用をしない）。

全代替案が現行モデルに劣ると評価された場合、**「現行で妥当（深化不要）」が正式な結論**となる。変更のための変更をしない——深化の成功は変更量ではなく説明力で測る。

**✅承認ゲート**: Before/After対比と各代替案の採点を提示し、採否は人間が判断する。非対話環境では採否判断をせず、確認事項として列挙する。

### STEP 5: 再構成

- 採用案でモデルを再記述する。
- 新概念の定義（名前・意味・ルール）、旧→新の対応表（分割・統合のマッピング）、ルールの再配置（どのルールがどの概念の責務になったか）を作る。

## 反証ラウンド（必須・スキップ禁止）

以下すべてに明示的に答える。1つでも該当したらSTEP 2〜3へ戻って修正する。検討の結果「修正せず維持」と判断した場合も、該当の事実と維持の理由を「判断根拠」に記録する（無言スルーのみ禁止）。

1. 新概念は既存概念の「改名」にすぎないのではないか（何の例外・フラグが消えたか具体的に言えなければ改名）。
2. 抽象化しすぎて現場の言葉から遊離していないか（そのドメインの人に通じるか）。
3. 深化の根拠がAIの一般知識であって、このドメインの素材に遡れないのではないか。
4. 「その他」「汎用」「共通」という名の概念を新設していないか。
5. 壊した前提（捨てた旧スキーマ）を記録に残したか。

## 出力契約（この形式以外で出力しない）

1. **Before/Afterモデル対比**: 概念定義表＋可能ならMermaid `classDiagram`。
2. **脱構築ラウンド記録**: 中心概念／代替案3つ以上／4基準の採点／推奨。（採点と推奨の根拠はここに書き、項目1には構造と対応表のみを書く——重複を避ける）
3. **消えた例外・フラグの一覧**: 深化の証拠。Beforeで例外処理だったものがAfterでどう自然に説明されるかを対で示す。
4. **判断根拠・確認事項・保留事項**: 確認事項（人間への質問。非対話環境の承認ゲートはここに列挙する）と保留事項（実行者が置いた仮定）を区別する。壊した前提の記録と反証ラウンドの記録を含む。
5. **ネクストアクション**: → mino-contract-driven-coding で実装すべき概念、→ mino-context-discovery の再実行が必要な境界、を指名する。（前工程への再実行は人間の明示指示があるときのみ。自動連鎖では戻らない）

## アンチパターンガード

- **賢く見える造語への逃避**: 深いモデルの名前は現場の言葉に根ざす。誰も使わない造語は深化ではなく難読化。
- **「全部作り直し」提案**: 移行パスのない深化は絵に描いた餅。旧→新の対応表がない再構成は提出しない。
- **深いモデル＝複雑なモデルという誤解**: 深いモデルは例外が消えてむしろ単純になる。代替案が既存より説明変数を増やしているなら退化を疑う。
- **採点なしの直感採用**: STEP4の4基準採点を経ずに代替案を採用しない。

## 品質基準（Doneの定義）

- 中心概念ごとに代替案が3つ以上、4基準で採点されている。
- 採用案はSTEP1で記録した例外・フラグを最低1つ具体的に消している（消えた数がゼロなら深化ではなく改名を疑う）。結論が「現行で妥当」の場合はこの限りではなく、全代替案の採点記録が根拠となる。
- 反証ラウンド5問すべてに回答があり、該当・維持の判断が理由付きで記録されている。
- 出力契約の5点が揃い、旧→新の対応表からモデルの移行経路が読み取れる。
