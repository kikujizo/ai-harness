# ループエンジニアリングの原則

エージェントに毎回プロンプトする代わりに、**エージェントを自律的に起動・反復・停止させるシステム自体**を設計する。
プロンプト → コンテキスト → ハーネス（ルール・ツール・フック）の上に積み上がる一階層上の規律で、
「その足場を いつ・誰が・何回・どこまで 回すか」を決める。

このファイルは全ループ設計の基盤原則を定める。新しいループを作るときは
`.agents/skills/loop-design/SKILL.md` の手順に従い、結果を
`docs/loop-ledger.md` の形式で台帳に登録する。

## 1. ループの6構成要素

すべてのループはこの6つを備える。埋まらない要素があるループは動かさない。

| 要素 | 内容 | 汎用ハーネスでの実装例 |
|---|---|---|
| ①自動化トリガー | スケジュール・イベントでループを起動する | Issueラベル / 手動起動 / cron監視 |
| ②並列隔離 | 作業空間を分離して競合を防ぐ | featureブランチ + 同時実行制御 |
| ③知識バンドル | プロジェクト知識を反復間で持ち回る | AGENTS.md + skills/ + テンプレ |
| ④外部連携 | 既存ツールと接続する | CLI / MCP / 通知Webhook |
| ⑤分業設計（Maker-Checker） | 生成役と検証役を分離する。自己採点は構造的に甘い | レビュアー独立（実装者≠レビュアー）+ verdict契約 |
| ⑥状態の外部化 | セッション間の文脈をファイル・ラベルに永続化する | ラベル状態機械 + Issue/PR本文（SSOT） |

## 2. 4ハードガード（すべてのループに必須）

このハーネスにおける**無進展検出を定義する唯一の正本はここ**。他ファイルはこの定義を参照する。

1. **反復上限** — 何回で打ち切るかを数値で固定する（例: 修正ラウンド最大2回、フォールバック各1回、
   recursive-writing最大2周／recursive-review再照合最大1回）。
2. **時間上限** — 全ステップに timeout を置く。1反復・全体それぞれに数値を決める（長時間ハングは実害が最大）。
3. **無進展検出** — 直近Nターンで**差分（新しいコミット・新しい指摘・状態遷移のいずれか）**がなければ停止する。
   「同じ指摘 → 同じ修正失敗」の空転は、反復上限だけでは枠の分だけ無駄に燃やす。
   実装形: 「前の反復と同じ結果（同一の逸脱リスト・同一のエラー）なら即停止し、構造的問題として人間へ返す」。
4. **予算キャップ** — ループ単位でトークン・実行時間を計測し上限を置く。
   放置されたループが暴走して高額を消費した外部事例が複数ある。サブスク枠でも「枠の食い潰し」として同じ構造。

台帳にないループは存在してはならない（野良ループ＝暴走リスク）。台帳登録は不可逆カテゴリ③（パイプライン自己変更）の一部として扱う。

## 3. 自律度レベルとラチェット

| レベル | 内容 | 例 |
|---|---|---|
| L1 報告のみ | 検出・提案までで実行しない | プロセスレビュー、滞留の通知 |
| L2 検証付き実行 | 実行するが、独立検証を通過したものだけ反映する | 通常リスクの自動レーン（レビュー + 検証green → 反映） |
| L3 無人実行 | 検証も機械化し、人間は事後確認する | 自動マージ（検証green ∧ レビュー通過 ∧ 非高リスク） |

**ラチェット構造（上げるのはゆっくり・証拠ベース、下げるのは即時）:**

- **開始**: 新しい種類のループは必ず **L1** で始める。
- **昇格**: 直近10回の実績で誤動作ゼロ、かつ停止時の復旧手順が文書化されている → 1段階上げてよい。根拠は実績（誤動作率・ゼロタッチ成功率）であって感想ではない。
- **降格**: インシデント1回（不可逆4カテゴリに触れた／人間が意図しない不可逆操作が起きた）→ **即1段階下げる**。原因分析とループ定義の修正が済むまで戻さない。不可逆4カテゴリの定義はルートの`AGENTS.md`（リスク分類）が正本。
- **メタループの例外**: ループがループ自身やルール（AGENTS.md・基準・prompts）を書き換えるループは、可逆工程（設計・実装・独立レビュー・PR作成）までAIが進めてよいが、**発効点（merge・設定反映）の人間approve/denyを恒久的に必須**とする（L2/L3の自動反映へ昇格させない）。ループがループを書き換えるのは不可逆カテゴリ③の再帰版。
- 昇格・降格は必ず Decision Log（`docs/templates.md`）に残し、「取り消し手順」欄を必ず埋める。操作面の運用は `docs/risk-dial.md`。

## 4. 研究的裏付け（簡潔）

なぜ「外部フィードバック」「分業」「毎回リセット」が必須かの実証根拠。詳細な失敗様式の照合は [failure-map.md](failure-map.md)。

- **外部フィードバックなしに自己修正できない** — 信頼できる外部信号（テスト・ツール検証・別モデル評価）があるタスクでのみ自己修正が機能し、純粋な内省だけでは修正後の品質がむしろ下がることが繰り返し報告されている（[TACL: When Can LLMs Actually Correct Their Own Mistakes?](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00713/125177/)）。だから基準ファイル・テスト・独立レビュアーがループの前提部品になる。自分の感想で自分を採点させるループは回しても収束しない。
- **Evaluator-Optimizer** — 生成LLMと評価LLMを分けたループは「明確な評価基準があり、反復が測定可能な改善を生む場合」にのみ有効（[Anthropic: Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)）。基準が曖昧なタスクでループを回すのはトークンの燃焼にしかならない。
- **Context Rot（文脈の腐敗）** — 入力が長くなるほど性能が非線形に（崖状に）劣化することが多数のフロンティアモデルで実測されている。含意: 会話を頑張って続けるより、各反復を**短い新鮮な文脈**で始め、進捗は会話履歴ではなくファイル・状態に置く方が強い。
- **Ralphループ** — 同じアンカーファイルを反復ごとにリセットしながら与え続ける原始的手法。進捗はディスクとgitに置く。教訓: **アンカーファイル（AGENTS.md・仕様・Issue本文）の品質が自律ループの品質の天井**になる。1反復1作業単位を守る。

## 5. 人間側の3つの警告

1. **検証責任は残る** — ループの出力は「提案」であって「証明」ではない。L3でも事後サンプリング確認を捨てない。
2. **理解債務** — ゼロタッチが続くほどコードベースへの理解ギャップが拡大する。対策: マージ要約 + 定期的な「今週マージされたもの」の人間レビュー。
3. **認知的降伏** — 「考えたくないからループに投げる」が始まったら危険信号。意図の入力（要望→要件）は人間の仕事として死守する。
