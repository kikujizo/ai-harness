# Skill発動テレメトリ — 発動実績の計測方法と実測記録

Skillは「作った本数」ではなく「発動した回数と、発動が行動を変えたか」で評価する。
本書はその計測方法（ログからの機械集計＋目視判定）と、orchestrate手順4「委譲時のSkill名指しルール」の根拠となった実測記録を保持する。

## 計測方法

### データソース

| ソース | 場所 | 保持 |
|---|---|---|
| Claude Code セッションログ（JSONL） | `~/.claude/projects/<project>/<sessionID>.jsonl` | 約30日で自動削除 |
| セッションのMarkdownエクスポート | 各ワークスペース側の履歴フォルダ（あれば。JSONLより長期） | 運用次第 |
| サブエージェントのログ | `~/.claude/projects/<project>/<sessionID>/subagents/agent-*.jsonl` | 親と同様 |

### 発動マーカー

- Skillツール経由: `Launching skill: <skill名>`（ツール結果）／ `"skill":"<skill名>"`（ツール入力）
- スラッシュコマンド経由: `<command-name>/<name>` と `<command-message>`

### 判定の規律

1. **偽陽性を除外する**: 会話中でSkillを引用・議論した箇所もマーカーにヒットする。件数確定時は各ヒットの前後を目視する
2. **陽性対照で測定器を検証する**: 「発動ゼロ」を主張する前に、同じログにRead/Write等のツール記録が存在することを確認する（記録機構が動いている中でSkillだけ無い＝真のゼロ）
3. **効果は自己申告でなくログで判定する**: エージェントの「Skillに従いました」は採用しない。発動直後に規定の行動変化（委譲・素材固定・署名等）がログ上で起きたかを見る

## 実測記録

### 2026-07-17: 3週間ベースライン（Vault76_Cloud）

- 期間6/26〜7/17でSkill発動14件。全件が適合発動（誤発動ゼロ）、発動直後の行動変化も全件で確認
- 一方、ユーザーの明示フレーズまたはルーティング表に載っていないSkill（内的状態トリガーの認知科学系）は**自発発動ゼロ**
- 教訓: 「状況の名詞列挙」型トリガーは発動を生まない。「この言葉を書く直前」「このツールを呼ぶ直前」という瞬間特定型に書き換える

### 2026-07-17: 下位モデル自発発動プローブ（名指しルールの根拠）

- haiku/sonnetのサブエージェント3体に、Skillトリガーを踏む自然なマイクロタスクを付与（プロンプトにSkillの存在は書かない）→ **自発発動 0/3**（transcriptマーカーで実測、陽性対照確認済み）
- Skill一覧の可視性・Skillツールの保有は別プローブ（haiku）で確認済み。つまり「見えているのに取りに行かない」
- 帰結: 委譲プロンプトでは従うべきSkillを名指しする（orchestrate手順4）。名指しのコストは1行、省略のコストは規律なし成果物であり、非対称

### 限界

- プローブはn=3・単一セッション・2モデル。機構（なぜ取りに行かないか）は未解明の推測
- 名指しルールは保守側に倒した実務規則として採用しており、反例（自発発動する条件）が観測されたら本書に追記して緩和を検討する
