# mino-contract-driven-coding — 契約による設計から生成する実装

ドメインモデルと要件から、契約による設計（Design by Contract）で「契約表」を先に明文化し、それからコードを生成する実装スキル。契約表を仕様の正本とすることで、何度実行しても要件を満たすコードを再現できる（コードだけでなく設計書の再生成にも使える）。原則は言語非依存であり、例示にはTypeScriptを使うが任意の言語に適用できる。

理論的裏付けは契約による設計（Bertrand Meyer）とドメイン駆動設計における完全コンストラクタ・値オブジェクト・集約の規律。ミノ駆動氏の公開講演にインスパイアされた非公認の再構築（出典は本スキル群のcore/mino-core-principles.md）。

## 使いどき / 使わないとき

**使う**: ドメインモデル・集約をコードに落とすとき／AIにコード生成させると毎回違うものが出て困っているとき／ビジネスルールをコードで確実に守らせたいとき／設計書とコードの整合を保ちたいとき。

**使わない**: モデル自体がまだ曖昧なとき（→ mino-model-deepening / mino-event-storming が先）／既存コードのレビュー（→ mino-changeability-review）／UI・インフラ層の実装（このスキルの対象はドメイン層）。

## コア原則（mino-*共通）

1. **言語ゲーム**: 言葉の意味は定義ではなく、文脈と目的の中での使われ方で決まる。定義論争から入らず「誰が・何の目的で・どう使うか」から入る。
2. **四つ組**: 意味理解の最小確認単位は「アクター／目的／文脈／ルール」。この4つが揃わない用語理解は未完成として扱う。
3. **目的駆動**: 構造は目的に従う。境界付けられたコンテキストとは「大粒度の目的」である。迷ったら「なぜ？」で上位目的へ遡る。
4. **スキーマ理論**: 人もAIも既存の認知枠組みで対象を見る。既知の概念への執着がモデルの成長を止める（「悪魔」と呼び続ける限り病原菌は発見できない）。
5. **脱構築**: 「みんなは○○と呼んでいるが、正体は××ではないか？」と問い続ける。固定化した言葉に構造を引っ張らせない。
6. **均質性制御**: AIの出力は学習内容の平均＝均質に収斂する。だから区分（スキル）を切り、区分内では出力契約で均質化させ、区分間の濃淡・軽重の設計は人間が握る。
7. **人間pilot原則**: 問題定義フェーズは人間がpilot、AIはcopilot。AIは問い・選択肢・根拠を出し、決定はアクター（人間）に委ねる。
8. **根拠の遡行可能性**: すべての結論に「どの素材・発言から導いたか」を添える。遡れない結論は仮説として明示する。

## 用語の定義（記入基準）

| 用語 | 一言定義 | 記入例 |
|---|---|---|
| 事前条件 | 呼び出す側が操作を呼ぶ前に満たしておくべき条件 | 数量は1以上200以下の整数であること |
| 事後条件 | 操作が実行後に保証する結果 | 在庫数は指定数だけ確実に減少している |
| 不変条件 | どんな操作の前後でも常に真であり続ける条件 | 在庫数は常に0以上200以下 |
| ドメインモデル完全性 | 不正な状態のオブジェクトがそもそも作れず、外部から勝手に破壊されない設計規律の総称 | 完全コンストラクタ・不変（immutable）・集約ルート経由の変更 |
| 集約ルート | 集約の唯一の入口となるオブジェクト。内部の子オブジェクトへの変更は必ずここを経由する | 注文（Order）が注文明細（OrderLine）の唯一の変更窓口 |

検証責任の方針: Meyer原典のDbCは「事前条件は呼び出し側の義務（供給側は再検証しない）」だが、本スキルは意図的に「不正な値はオブジェクト生成・操作の境界で拒否する」（完全コンストラクタ／always-valid）を既定とする。AIが生成するコードでは呼び出し側の規律を信頼できないため、**契約の記法**にDbCを、**検証の配置**に境界検証を採る折衷である。

## インプット

必要なもの: 実装対象のドメインモデル・集約。mino-model-deepening / mino-event-storming の出力（集約候補＋不変条件）があれば理想的なインプット。

足りないとき: 推測で埋めず、最低限次を人間に確認する（一度に3問まで）: ①対象概念とその境界（集約の範囲はどこまでか） ②守るべきルール（値の範囲・単位・null許容・状態遷移の可否を数値・具体語で） ③使われ方（誰が・どういう操作として呼ぶか）。

対話できる環境では、✅承認ゲートで出力を止めて人間の返答を待つ（返答なしに次のSTEPへ進まない）。対話できない実行環境（バッチ実行・非対話エージェント）では、仮定したことを出力の「保留事項」に明記して続行する。承認ゲートは「確認待ち」ではなく「確認事項の列挙」として出力する。

## 手順

### STEP 1: 要件とモデルの受領

- 実装対象の概念・ルール・ユースケースを確認する。
- mino-model-deepening / mino-event-storming の出力（集約候補＋不変条件）があれば理想的なインプットとしてそのまま使う。
- なければ、インプット節の質問で最低限「対象概念・守るべきルール・使われ方」を確認する。

### STEP 2: 契約表の作成（コードより先。このスキルの心臓部）

- 各ドメイン概念・各操作について、事前条件（呼び出す側が守ること）／事後条件（操作が保証すること）／不変条件（どんな操作の後でも常に真であること）を表で明文化する。
- 値の範囲・単位・null許容・状態遷移の可否・境界値を必ず数値で書く（例: 数量は1〜200の整数。ミノ駆動氏の講演のQuantityクラス例のように、プリミティブ型に許される範囲を型自体の責務として書き下す）。
- 「なんとなく妥当」で済ませず、最小値・最大値・境界±1・null/未設定の扱いまで埋める。埋まらない項目は次STEPへ進まず、インプットに立ち戻って確認する。
- 範囲・条件を外れた場合の挙動も必須記入項目とする。既定は「拒否（エラー）」。丸め（clamp）や黙認を選ぶ場合は理由を契約表に書く。

**✅承認ゲートG1**: 契約表を提示して合意を取る。**契約表が以後の仕様の正本になる**（コードと設計書はここから再生成可能になる）。

### STEP 3: インターフェイス設計（実装より先）

- 目的駆動の名前で公開インターフェイスを定義する（実装なし）。
- 1インターフェイス＝1目的。目的が違うなら処理が似ていても分ける（例: 通常会員の年間ポイント付与とゴールド会員の年間ポイント付与は、計算式が似ていても別クラス。目的の分岐を1クラス内のif分岐にしない）。
- 命名は「何をするか（動詞＋目的語）」ではなく「何のためか（目的）」が伝わる名前を優先する。

### STEP 4: 実装生成（ドメインモデル完全性の規律）

契約表とインターフェイス定義に従い実装する。以下5点の規律を必ず守る。

- (a) 不変条件はコンストラクタ／ファクトリで強制し、不正な状態のオブジェクトをそもそも作れなくする（完全コンストラクタ）。
- (b) setter禁止・不変（immutable）優先。変更は新インスタンス返却で表現する。
- (c) ロジックはそのデータを持つクラスの中に置く（ドメイン知識をドメイン層の外に漏らさない）。
- (d) 集約内部の変更は必ず集約ルート経由。子オブジェクトを外から直接いじらせない。
- (e) ドメイン値はプリミティブ型のまま運ばず値オブジェクトに包む。ただし振る舞いを持たない識別子（ID）は等価性のみの軽量な型でよい。

### STEP 5: 契約検証テストの導出

- 契約表の各行から機械的にテストを導出する。事前条件違反→拒否されること／事後条件→保証が成立すること／不変条件→どの操作列の後も維持されることを検証する。
- 境界値（最小・最大・±1）は必須。契約表に書かれていない境界値のテストを追加しない（テストは契約表の翻訳であり、独自解釈を持ち込まない）。
- テスト対象は公開インターフェイスのみ。内部専用メソッドの実装詳細はテストしない。
- テスト検討中に契約表にない懸念（操作の相互作用・冪等性・並行実行など）を発見したら、テストを勝手に足さず**契約表へ行を追加（差し戻し）してから翻訳する**。契約表の網羅性がそのまま品質の上限になることを忘れない。

**✅承認ゲートG2**: 実装＋テストを提示する。コードを読まない承認者のために、契約表の各行について「この条件のときこう動く」を平易な言葉で1行ずつ要約した充足リストを必ず添える（承認の判断材料はコードではなくこのリスト）。

### STEP 6: 再現性の突合

- 契約表の全行について「実装のどこで守られ、テストのどこで検証されるか」の突合表を作る。契約表1行に複数テストケース（境界値群）をまとめて対応付けてよい。
- 担保されていない行（＝暗黙仕様）をゼロにする。ゼロにできない場合は出力契約5の確認事項として明示し、次サイクルへ持ち越す。

## 反証ラウンド（必須・スキップ禁止）

以下5問すべてに明示的に答える。1つでも該当したらSTEP 2（契約表）へ戻って修正する。検討の結果「修正せず維持」と判断した場合も、該当の事実と維持の理由を出力契約5の「判断根拠」に記録する（無言スルーのみ禁止）。

1. 契約表に書けなかった要件が残っていないか（コードにだけ存在する暗黙仕様は再現性を壊す）。
2. プリミティブ型執着がないか（int/string/floatのままドメインの値を運んでいる箇所）。
3. getterでデータを取り出して外側で判断・計算していないか（Tell, Don't Ask違反＝カプセル化破りの典型）。
4. 集約ルートを迂回して内部オブジェクトを直接操作する経路がないか。
5. 目的の違う処理を「似ているから」と共通化して混ぜていないか。（判断材料が乏しいときは分離を既定とする。後から統合するのは容易だが、混ざったものの分離は困難）

## 出力契約（この形式以外で出力しない）

1. **契約表**: 概念・操作／事前条件／事後条件／不変条件。境界値は数値で書く。
2. **公開インターフェイス定義**: 目的駆動の名前。実装なし。
3. **実装コード**: ドメインモデル完全性の規律（STEP 4の(a)〜(e)）に従う。
4. **契約検証テスト**: 契約表の行と1対1対応。
5. **突合表＋判断根拠・確認事項・保留事項**: 契約表の行／実装箇所／テスト箇所の対応表。未担保行があれば明示する。確認事項（人間への質問。非対話環境の承認ゲートG1・G2はここに列挙する）と保留事項（実行者が置いた仮定）は性質が違うため区別して書く。反証ラウンドの該当・修正・維持の記録もここに含める。

## アンチパターンガード

- **コードを書いてから契約を後付けする**: 順序逆転は再現性を生まない。契約表なしにコードから書き始めない。
- **貧血ドメインモデル**: データだけのクラス＋外部のロジッククラス、という分離を作らない。
- **「とりあえずnullチェック」で契約を代替する**: 契約は範囲・単位・遷移まで含む。nullチェック1つでは不変条件の代わりにならない。
- **テストが実装の写経になっている**: テストは契約表から導出する。実装から導出しない（実装のバグごとテスト化してしまう典型的失敗）。
- **集約ルートの意味を理解しないgetter大公開**: 内部オブジェクトを外に晒し、外側で変更・判断させない。

## 品質基準（Doneの定義）

- 契約表の全行が、実装のどこかとテストのどこかに対応している（突合表に空欄がない、または空欄が確認事項として明示されている）。
- 不正な状態のオブジェクトを生成する経路が存在しない（完全コンストラクタで拒否されることがテストで示されている）。
- 境界値（最小・最大・±1）のテストが契約表の数値と1対1で対応している。
- 反証ラウンドの5問すべてに明示的な回答があり、「該当したが維持」の場合も理由が判断根拠に記録されている。
- 出力契約の5点が揃っている。
