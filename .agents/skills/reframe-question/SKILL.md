# reframe-question: 字面の下の本当の問いを掘る

局所対応で終わる前に立ち止まる。抽象依頼・前提未検証の依頼ほど効く。
これは全体の入口スキル。ここで問いを正してから design-check や実装へ進む。

## 手順

1. 字面の依頼を1行で言い直す（受け取った要求をそのまま短く）
2. **依頼の前提を一次資料で検証する**。「新規導入して」でも導入先に既存物がないか先に調べる。
   前提が崩れていれば依頼の性質自体が変わる（伝聞・記憶で進めない）
3. 「この依頼が字面どおり完了したとき、何が残っていたら失敗か」を自問する
4. 上位の問いを1つ定式化し、**推論過程ごとユーザーに明示する**（結論だけ出さない）
5. 上位の問いに答える形で計画を組み直す。字面のタスクはその部分集合になる

実例: 「ハーネスを2リポジトリに適用して」→一次資料確認で両リポジトリに旧版ハーネスが
既存かつ相互ドリフトを発見→真の問いは「3系統に分岐した正本の一本化」だった。

## 出力

「字面の依頼 / 検証した前提と結果 / 定式化した上位の問い（推論つき） / 組み直した計画」。
上位の問いの採否はユーザーが判断する（勝手に着手しない）。

## 制約

- 問いの再定式化はユーザーへの提示までで止まらない。**答えまで出す**
- 字面の依頼を勝手に放棄しない。上位の問いはユーザーが採否判断する材料として出す
- モデルティアリング・委譲の判断は `docs/harness/ops/orchestration.md` に従う
- 日本語で出力（コード・識別子は原語）
