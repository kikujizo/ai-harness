# mino-skills 設計思想解説

## 1. この文書は何か

これは mino-skills スイートの**理論的背景と出典**を解説する文書である。SKILL.md本文（各スキルのコア原則節・出典行）は実行時にAIが参照するが、この文書はスキル実行時には読まれない。あくまで**人間の読者**——スイートを導入・検証・改訂する人——が「なぜこの手順・この語彙になっているか」を理解するための解説である。

各SKILL.mdの記述は簡潔さを優先しており、理論の出典や採用理由の背景説明は割愛されている。本文書はその割愛分を補い、6スキル共通の思想的基盤を1か所にまとめる。

## 2. 経緯と性格

mino-skillsは、ミノ駆動氏（仙塲大也氏、『良いコード/悪いコードで学ぶ設計入門』著者）がX（旧Twitter）や公開講演で存在を明かした、DMM社内製・非公開のagent skill群にインスパイアされて作られた。氏は「イベントストーミングもドメインモデリングもagent skill化している」と公言しているが、そのスキル本体（プロンプト文面・出力契約の詳細）は非公開である。

したがって本スイートは、氏が公開したスライド・ポストという**公開情報のみ**から、外部の人間が推定して再構築したものである。**本人非公認のオマージュであり、複製ではない**。何より、再現を試みたのは「プロンプトの文面」そのものではなく、氏が語った**「コア哲学×工程別モジュール」という構造**（後述のモディフィウス型アーキテクチャ）である。同じ構造の上に、公開情報から読み取れる理論的裏付け（言語ゲーム・スキーマ理論・契約による設計等）を独自に肉付けして手順化した。氏の実際のスキルとは中身が異なっている可能性が高いことを前提に扱ってほしい。

技術的な実装形式は Agent Skills オープン標準（agentskills.io）に準拠しており、Cursor・Codex・Claude Codeなど複数環境で同じSKILL.mdを共有できる。ChatGPT向けには本ディレクトリの`adapters/chatgpt/`に単一ファイル変換版を用意している。

## 3. 6スキルとパイプライン

6スキルには正準実行順があり、上流の出力が下流のインプットになるように設計されている。

| # | スキル | 一言役割 |
|---|---|---|
| 1 | `mino-socratic-requirements` | ソクラテス問答×目的の言語化による要求定義 |
| 2 | `mino-context-discovery` | 言語ゲーム×四つ組による境界発見 |
| 3 | `mino-event-storming` | イベントストーミングによる業務時系列分析→ドメインモデリングへの橋渡し |
| 4 | `mino-model-deepening` | 脱構築×スキーマ理論によるモデル深化 |
| 5 | `mino-contract-driven-coding` | 契約による設計×ドメインモデル完全性による実装 |
| 6 | `mino-changeability-review` | バグサーチャー×DbCテストによる変更容易性レビュー |

正準実行順は socratic-requirements → context-discovery → event-storming → model-deepening → contract-driven-coding → changeability-review。ネクストアクションが上流へ戻る指名（例: model-deepeningからcontext-discoveryへの再実行）は、人間の明示指示があるときのみ従う設計になっている（スキル同士が自動的に行ったり来たりする「ピンポン」を防ぐため）。

## 4. 理論の柱

各スキルのSKILL.mdに複製されている「コア原則」8項目の背景を、テーマ別に説明する。

### 4.1 言語ゲーム（ウィトゲンシュタイン）と四つ組

後期ウィトゲンシュタインの言語ゲーム論は、「言葉の意味は辞書的な定義ではなく、その言葉が使われる文脈と目的の中での用法によって決まる」とする考え方である。mino-skillsはこれを、ドメイン分析の出発点に据える。同じ「注文」という語でも、営業担当と物流担当では指しているものも従うべきルールも違いうる——この違いを定義論争（「注文とは何か」を辞書的に決めようとする不毛な議論）で解決しようとせず、「誰が・何の目的で・どう使うか」という使用の実態から出発する。

この使用実態を確認するための最小単位が「四つ組」（アクター／目的／文脈／ルール）である。ある用語の理解は、この4要素が揃って初めて完成したとみなす。**どのスキルで使うか**: 主に `mino-context-discovery`（用語×文脈のマトリクスを四つ組で埋める）。他スキルでも「目的」の確認・遡及に繰り返し使われる。

### 4.2 スキーマ理論（認知科学）

認知科学のスキーマ理論は、「人（そしてAIも）は生の対象を見ているのではなく、既存の認知の枠組み（スキーマ）を通して対象を解釈している」とする。この枠組みは一度定着すると自己強化的になり、枠組みに合わない事実を「例外」として処理し続けてしまう。

ミノ駆動氏はこれを「病原菌を『悪魔』と呼び続ける限り、病原菌は見つからない」という比喩で語っている。呼び名（スキーマ）が対象の理解を先に決めてしまい、その呼び名を疑わない限り、対象の正体には永遠にたどり着けないという意味である。増え続けるフラグ・種別コード・「その他」区分・例外処理は、この意味で「既存スキーマが現実に合わなくなっている兆候」として扱う。**どのスキルで使うか**: 主に `mino-model-deepening`（STEP2のスキーマ検出＝「悪魔候補」の特定）。

### 4.3 「脱構築」

> **注記**: ここでの「脱構築」は、ミノ駆動氏のX上の用法に倣った**通俗的な意味**（固定化した呼称を疑い、対象を捉え直すリフレーミングの技法）で使っている。ジャック・デリダの哲学用語としての厳密な脱構築（テクストの内的矛盾を通じた二項対立の解体、という厳密な方法論）を指すものではない。この点を混同しないよう注意されたい。

具体的には「みんなは○○と呼んでいるが、正体は××ではないか？」という問いを繰り返し、固定化した呼び名に構造を引っ張らせないようにする作業を指す。スキーマ理論（4.2）とセットで機能する——スキーマによって固定された呼び名を、脱構築の問いによって強制的に疑うことで、初めて枠組みの外に出られる。**どのスキルで使うか**: `mino-model-deepening`（STEP3の脱構築ラウンド。時間・目的・統合・出来事・役割という5つの観点で代替案を強制生成する）。

### 4.4 目的-目標-手段の3層と「境界付けられたコンテキスト＝大粒度の目的」

ミノ駆動氏が講演で説く区別として、「目的」（実現したい価値・状態）、「目標」（目的達成の指標）、「手段」（目標達成のための具体策）の3層がある。持ち込まれる要望の多くは、実際には「手段」の段階のものであり、それを目的であるかのように扱ってしまうと、より良い代替手段を検討する余地を失う。`mino-socratic-requirements`の「手段固着チェック」はこの区別に基づく。

この目的駆動の考え方をドメインモデリングに拡張したのが、「境界付けられたコンテキスト（DDDの戦略設計における概念）とは、大粒度の目的である」という定式化である。コンテキストの境界は、組織図やDBのテーブル構成やUI画面をなぞって引くものではなく、「目的が変わる場所」で引く。目的が変わるところで言葉の意味とルールも変わる、という言語ゲームの帰結でもある。**どのスキルで使うか**: `mino-socratic-requirements`（目的遡及の問い）、`mino-context-discovery`（STEP3の境界画定基準）。

### 4.5 契約による設計

契約による設計（Design by Contract、Bertrand Meyerが提唱）は、操作を「事前条件（呼び出す側が満たすべき条件）」「事後条件（操作が保証する結果）」「不変条件（常に真であり続ける条件）」の3種の契約として明文化し、それを仕様の正本とする考え方である。mino-skillsでは、この契約表を先に作ってからコード・テストを生成する順序を徹底することで、AIによるコード生成のばらつき（同じ依頼でも毎回違う実装が出る問題）を抑える。

> **注記**: 本スイートは契約の**記法**にDbCを借りているが、契約の**検証をどこに配置するか**についてはMeyerの原典（事前条件の検証は呼び出し側の責任であり、供給側は再検証しない）を採らない。代わりに、完全コンストラクタ／always-valid（不正な値をオブジェクトの生成・操作の境界で拒否する）を既定にする**折衷**を採用している。理由は、AIが生成するコードでは「呼び出し側が契約を守っている」という前提そのものを信頼できないためである。境界で拒否する設計の方が、AI生成コードの実務上の安全性が高い。

**どのスキルで使うか**: `mino-contract-driven-coding`（契約表の作成が手順の心臓部）、`mino-changeability-review`（観点(b)「契約」で契約欠如を検出し、追加すべき契約テストを提案する）。

### 4.6 均質性制御

ミノ駆動氏がXで語っている論点として、「AIの出力は学習内容の平均、すなわち均質なものに収斂しやすい」という指摘がある。多様さ・濃淡・軽重を求める作業をAI任せにすると、どの部分も同じような粒度・同じようなトーンで均されてしまう。

mino-skillsはこれに対し、「1スキル＝1区分」という設計で応じる。区分（スキル）を切ることで、区分の内側では出力契約という固定フォーマットにあえて均質化させ（同じスキルなら誰が使っても同程度の品質・構造の出力になる）、その代わり区分と区分の間の濃淡・軽重——どこを厚く検討し、どこを軽く流すか——の設計は人間が握る。均質化を全面的に排除するのではなく、均質化してよい範囲を区分によって明示的に区切る、という制御の仕方である。**どのスキルで使うか**: 全スキル共通（各SKILL.mdの「出力契約」節がこの均質化装置にあたる）。

### 4.7 人間pilot / AI copilot

問題定義フェーズ（何を解くべきかを決める段階）では、人間が操縦者（pilot）でありAIは副操縦者（copilot）である、という役割分担の原則。AIは問い・選択肢・根拠を提示するところまでを担当し、決定そのものはアクター（人間）に委ねる。`mino-socratic-requirements`が問答フェーズで解決策を一切出さないこと、各スキルの「✅承認ゲート」で対話可能な環境では出力を止めて人間の返答を待つこと、いずれもこの原則の具体化である。**どのスキルで使うか**: 全スキル共通、特に上流4本（socratic-requirements〜model-deepening）の承認ゲート設計。

## 5. 設計上のトレードオフ（正直な限界）

推定再構築という性質上、いくつかのトレードオフを意図的に許容している。

- **コア原則の複製**: 各スキルが単体で（他ファイルなしに）動くことを優先する「自己完結原則」のため、8項目のコア原則を6つのSKILL.mdすべてに複製している。正本（SSOT）は`core/mino-core-principles.md`であり、改訂時はそこを直してから全スキルへ転記する必要がある——放置すると複製先が容易にドリフトする。
- **設計規律の二重化**: 「完全コンストラクタ」「setter禁止」「集約ルート経由の変更」等の設計規律が、`mino-contract-driven-coding`（実装時に守る規律として）と`mino-changeability-review`（レビュー時に検出する観点として）の2本に別の文言で重複している。これも自己完結原則を優先した意図的な二重化であり、規律の内容を変える際は両方を同時に更新する必要がある。
- **非対話モードでの問答スキルの格下げ**: `mino-socratic-requirements`の本質は「1回1問」の実際の往復にある。バッチ実行・非対話エージェント経由など、往復ができない環境ではこの手順の土台自体が崩れる。そのためこのスキルは非対話モード専用の代替手順を持つが、それでもAIの推定回答は人間本人の回答の代わりにはならない。非対話モードで作成した要求定義書は見出しに「（未承認ドラフト・非対話推定）」を明記し、人間の確認を経るまで後続スキルへ自動連鎖しない設計にしている。
- **契約表の網羅性が品質の上限**: `mino-contract-driven-coding`はテストを契約表から機械的に導出する設計であり、契約表に書かれていない懸念はテスト化されない（テスト中に発見した懸念は契約表への差し戻しを必須にしている）。裏を返せば、契約表が粗ければ生成されるコード・テストの品質もその粗さに引きずられる。契約表の網羅性そのものが、このスキルの成果物の品質上限になる。
