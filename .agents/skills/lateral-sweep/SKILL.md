# lateral-sweep: 1件を失敗クラスに広げて全域を掃く

指摘は1箇所で終わらせない。レビュアーが1箇所指摘したら、残りは自分で見つける。
assessment-first の横展開チェックから呼ばれることが多い。

## 手順

1. 指摘を**失敗クラス**に抽象化する（「何の欠如がこれを許したか」）。
   例: SSOT違反=正本宣言しつつ再掲 / 幻参照=実在しない手順への言及 /
   環境非対称=Windows製ゆえの欠陥 / 前提の暗黙化
2. クラスをgrep可能な特徴に落とし、対象全域を**読み取り専用**で機械走査する。
   大量走査は `orchestrate` 経由で下位モデルへ委譲可能（委譲機能がない環境では設計案のみ）
3. 検出物を三分する: **修正候補** / **受容**（理由を記録）/ **バックログ**
4. 失敗クラスが正本起因なら上流へ還流方針を示す（`docs/harness/knowledge/reflux.md`）。
   Issue/PRに全件・影響範囲・受容判断を記録し、**後続Issue候補**を提案する
5. **本Skillではファイル修正・PR作成・merge判断を行わない**。修正は承認済みIssueと
   実装担当の明示後の別作業とする

実例: レビュー指摘2件を4クラスに抽象化→正本37ファイルの機械点検で新規18件を検出→
Issueに修正候補・受容・バックログを記録し、後続Issueを起票候補として列挙した。

## 制約

- 走査は**読み取り専用**で行う（掃きながら壊さない）
- 無承認でファイル修正・PR作成・merge判断をしない
- 「修正しない」判断も必ず理由つきで記録する（無言のスキップ禁止）
- 記録はGitHubのIssue/PRへ（ルート`AGENTS.md`「GitHubドリヴン記録」）
- 日本語で出力（コード・識別子は原語）
