# コードレビュー基準（code-review-criteria）v1.0 — 較正期間中

> **これは何か**: AIが実装したコード変更（diff/PR）を `recursive-review` で照合するためのバイナリ基準（**第1層**）。
> ツール非依存 — Cursor / Codex / Claude Code / ChatGPT のどれに貼っても使える。**正本はこのファイル**（導入先リポジトリ・Vault側のコピーは参照/作業版）。
> **第2層との関係**: 本基準は「既に観測した失敗」を塞ぐ層。未知の失敗・品質問題（設計不適切・肥大化・防御の削除等）は [quality-lens.md](quality-lens.md)（第2層・集合知レンズ）が受け持ち、実案件で発火した指摘だけが本基準へ昇格する。
> **較正状態**: v1.0作成直後。最初の2〜3回の運用では判定を確認して較正する。**確定判断は人間**：
> 項目1〜6は人間が証拠（ログ・URL・チェックボックス）だけで確認でき、項目7〜10は第二AIが一次スクリーニングし、人間はその指摘の納得感を確認する。判定が割れたら直すのは判定ではなく**基準の書き方**。割れの原因は「基準の欠陥」か「チェッカーの誤判定」かを切り分けてレビュー記録（GitHubコメント）に残す。
> **由来の凡例**: [実測] = kikujizo配下リポジトリの直近PRで実際に起きたミス（2026-07-11調査）/ [暫定] = 一般的AIミス傾向由来。6ヶ月×が出なければ剪定候補（剪定の確定は「×ゼロ」に加えて「実測ミスの再発ゼロ」を人間が確認してから）。

**長さ中立**: 簡潔な成果物は冗長な成果物と同等以上に評価する。

## 基準（10項目・○/×/確認不能）

1. **実行証拠がある** [実測]
   観測手順: 変更コードを実際に実行した出力（テスト結果・CIの緑ログへのリンク・コマンド出力）がPR本文または完了報告に貼られているか見る。「動くはず」という記述のみなら×。
   **貼られているだけでは○にしない**: CIリンクは開いて「対象commitのSHAと一致する実行が緑」であることを確認する。コマンド出力は可能なら再実行して突合する（捏造・過去ログの使い回し対策）。
   （根拠: ai-dev-workflow で未実行のworkflow YAMLが7連続fix PRになった — [#3](https://github.com/kikujizo/ai-dev-workflow/pull/3)ほか）

2. **実行環境の前提を確認している** [実測]
   観測手順: コードが前提にする環境要素（OS・shellの種別/バージョン・文字コード・PATH/認証・権限）のうち、対象環境と食い違うと壊れるものについて、実環境で確認した記録があるか見る。Windows / PowerShell 5.1 / UTF-8(BOM) / OneDrive 配下は特に。
   （根拠: shell種別の迷走 [#8](https://github.com/kikujizo/ai-dev-workflow/pull/8)・[#10](https://github.com/kikujizo/ai-dev-workflow/pull/10)、Shift-JIS文字化け [#12](https://github.com/kikujizo/ai-dev-workflow/pull/12)）

3. **受け入れ条件と1対1で対応している** [実測]
   観測手順: 元Issue/依頼の受け入れ条件がPR本文（または報告）に逐条転記され、各条に「対応する変更箇所 or 証拠」が1対1で紐づいているか見る。条文の欠落・「まとめて対応済み」表記は×。
   受け入れ条件が明文化されていない依頼の場合: 照合の前に依頼文から条文化し、依頼者に見える形で提示してから当てる（条文がないまま○を付けない）。
   （根拠: 完了条件の狭い解釈で2段階修正 — dev-dashboard [#26](https://github.com/kikujizo/dev-dashboard/pull/26)→[#29](https://github.com/kikujizo/dev-dashboard/pull/29)）

4. **重複作業がない** [実測]
   観測手順: 同一・関連Issueに対する他AI/他者のopen PRがないことを確認した記録（`gh pr list`の結果等）があるか見る。
   （根拠: 同一Issueへ2系統のPRが並行 — slack-memory-bot [#143](https://github.com/kikujizo/slack-memory-bot/pull/143)/[#150](https://github.com/kikujizo/slack-memory-bot/pull/150)）

5. **必須フォーマットを満たしている** [実測]
   観測手順: そのリポジトリ/組織の必須フォーマット（PRテンプレ・記録者明記・Decision Log等の必須欄）とPR本文を突き合わせ、欠落ゼロか見る。埋まらない欄は「なし」と明記されていること。
   （根拠: 署名欠落・記載誤りを3ラウンド同一指摘 — ai-harness [#28](https://github.com/kikujizo/ai-harness/pull/28)）

6. **マージ反映の確認手順がある** [実測]
   観測手順: スタックPR・base変更・複数ブランチを含む場合、「merge後にmainへ変更が実在することを確認する手順」またはbase設定の正当性確認が示されているか見る。単独PR（base=main）は○扱い。
   （根拠: Merged表示だがmain未反映 — slack-memory-bot [#154](https://github.com/kikujizo/slack-memory-bot/pull/154)→修復[#156](https://github.com/kikujizo/slack-memory-bot/pull/156)）

7. **幻参照がない** [暫定]
   観測手順: diffが新規に参照する関数・ファイル・設定キー・API・CLIフラグを列挙し、それぞれ実在すること（grep・実行・公式ドキュメント）を確認する。1つでも実在しなければ×。

8. **波及漏れがない** [暫定]
   観測手順: 変更したシンボル/値/仕様について、同じ変更を要する他の箇所（呼び出し元・重複定義・設定ファイル・ドキュメント）を検索した形跡があり、残存箇所がないか見る。

9. **失敗パスを握りつぶしていない** [暫定]
   観測手順: 新設・変更されたエラー処理に、エラーを無言で捨てる形（空catch、`|| true`、単独の`-ErrorAction SilentlyContinue`等）がないか見る。意図的な握りつぶしには理由コメントがあれば○。

10. **スコープ逸脱がない** [暫定]
    観測手順: 元Issue/依頼の範囲外の変更（無関係ファイルの整形・リネーム・頼まれていない機能追加）がdiffに混ざっていないか見る。

## 判定規則

- 各項目は **○ / × / 確認不能** の3値。部分点・「7割OK」は禁止
- 対象に関係ない項目は「**対象外**」と明記する（黙って飛ばさない。対象外は○にも×にも数えない）
- **各判定には根拠1行**（実行ログ・grep結果・URL・ファイル:行）を添える。根拠のないチェックボックスだけの回答は無効
- ×は「項目番号 + ファイル:行（またはPR本文の該当箇所）」で名指しする
- **確認不能は×にしない**。確認不能として列挙し、確認方法（誰が何を見れば判定できるか）を添える。
  ただし **[実測]項目（1〜6）に確認不能が残る間は approve を出さない**（確認の宿題付きで request-changes として返す。重複PR・main反映・実行証拠という最悪ミスを未検査のまま通さないため）
- 基準にない問題に気づいたら「基準外の気づき」として分離して書く（本基準の改善候補。レビュー記録に残し、3回同内容で追記提案 — reflux ゲート②）
- **verdictの前に、全指摘へ三択を割り当てる**: 「今回修正（軽微はこれが既定）/ wontfix（理由1行）/ 後回し（**追跡Issue URL必須**）」。未割り当ての指摘を残したまま approve しない（「軽微な指摘はあるがマージ可」の禁止。追跡されない後回しは消滅する）

## 非エンジニアPMの読み方

コードの行は読まなくてよい。**「証拠」だけを見る**: 項目1〜6は実行ログ・チェックボックス・URLの実在で判定でき、コード知識は不要。項目7〜10はAIの照合結果（×の名指しと根拠）の納得感だけ確認すればよい。
