# mino-changeability-review — 変更容易性の設計バグサーチャー

コードの「動くかどうか」ではなく「安全に変更し続けられるか」を診る。将来の変更を高コスト・高リスクにする設計の病巣を検出し、影響度×修正コストで優先度を付け、改善スケッチと契約テストまで提案する。

ミノ駆動氏の公開講演（バグサーチャー・契約による設計・目的駆動アーキテクチャ）にインスパイアされた非公認の再構築（出典は本スキル群のcore/mino-core-principles.md）。

## 使いどき / 使わないとき

**使う**: PRレビュー・マージ前チェック／AIが生成したコードの品質確認／リファクタリング対象の選定／「このコード触るのが怖い」の正体を言語化したいとき。

**使わない**: 機能バグ・ロジック誤りの探索（動作の正しさは通常のテスト・QAで診る。このスキルの対象は設計の病巣）／モデル自体の見直し（→ mino-model-deepening）／新規実装（→ mino-contract-driven-coding）。

## コア原則（mino-*共通）

1. **言語ゲーム**: 言葉の意味は定義ではなく、文脈と目的の中での使われ方で決まる。定義論争から入らず「誰が・何の目的で・どう使うか」から入る。
2. **四つ組**: 意味理解の最小確認単位は「アクター／目的／文脈／ルール」。この4つが揃わない用語理解は未完成として扱う。
3. **目的駆動**: 構造は目的に従う。境界付けられたコンテキストとは「大粒度の目的」である。迷ったら「なぜ？」で上位目的へ遡る。
4. **スキーマ理論**: 人もAIも既存の認知枠組みで対象を見る。既知の概念への執着がモデルの成長を止める（「悪魔」と呼び続ける限り病原菌は発見できない）。
5. **脱構築**: 「みんなは○○と呼んでいるが、正体は××ではないか？」と問い続ける。固定化した言葉に構造を引っ張らせない。
6. **均質性制御**: AIの出力は学習内容の平均＝均質に収斂する。だから区分（スキル）を切り、区分内では出力契約で均質化させ、区分間の濃淡・軽重の設計は人間が握る。
7. **人間pilot原則**: 問題定義フェーズは人間がpilot、AIはcopilot。AIは問い・選択肢・根拠を出し、決定はアクター（人間）に委ねる。
8. **根拠の遡行可能性**: すべての結論に「どの素材・発言から導いたか」を添える。遡れない結論は仮説として明示する。

## インプット

必要なもの: レビュー対象のコード（ファイル・diff・PR）。あれば: そのコードが実現すべき要件・仕様、mino-contract-driven-coding の契約表（あれば契約との突合が最速のレビューになる）。

足りないとき: 対象の目的・変更予定の有無を人間に聞く（一度に3問まで）。

対話できる環境では、✅承認ゲートで出力を止めて人間の返答を待つ（返答なしに修正作業へ進まない）。対話できない実行環境（バッチ実行・非対話エージェント）では、仮定したこと（特にSTEP 1の目的推論）を出力の「保留事項」に明記して続行する。承認ゲートは「確認待ち」ではなく「確認事項の列挙」として出力する。

## 手順

### STEP 1: 目的の推論（レビューの前提づくり）

- 「このコード（ファイル・クラス）の目的は何か」を先に分析して明示する。
- 目的が読み取れない場合、それ自体を最重要の所見として記録する（目的不明のコードは変更容易性以前の問題）。
- 以後の所見はすべて「この目的にとって」という前提の上に立つ。目的の解釈が変われば所見も変わるため、推論の根拠（命名・コメント・呼び出し元）を記録する。

### STEP 2: 検出ラウンド（6観点・固定リスト）

全観点を必ず走査し、観点ごとに「該当あり／なし」を明示する（なしも書く。走査の証明）。

| 観点 | 見るもの |
|---|---|
| (a) カプセル化 | 生データの公開（public setter/getter大公開）、データだけのクラス＋外部のロジッククラス（貧血モデル・低凝集）、取り出して外で判断するTell Don't Ask違反 |
| (b) 契約 | 検証されない引数・戻り値、不正な状態で生成できるオブジェクト（コンストラクタが素通し）、nullの垂れ流し、範囲・単位が型で表現されないプリミティブ執着 |
| (c) 目的の混在 | 1クラス・1関数に複数の目的。種別フラグ・typeコードのif/switchで目的が分岐している箇所はクラス分離候補 |
| (d) 集約整合 | 集約ルートを迂回した内部コレクション・子オブジェクトの直接操作、整合性を一緒に保つべきデータがバラバラの場所で変更される構造 |
| (e) 命名 | 目的が伝わらない名前（Manager/Util/Common/Info/Data等）、意味の広すぎる名前、実装手段の名前（〜Flag、〜Str） |
| (f) 重複と発散 | 同一ビジネスルールの複数箇所実装。1つの仕様変更が多箇所同時修正になるshotgun surgery予備軍 |

### STEP 3: 優先度付け

- 各所見に「影響度（その箇所の変更頻度の見込み × 壊れたときの被害）」と「修正コスト」を高/中/低で付ける。変更予定が明示されている箇所は影響度を1段階引き上げる。
- 優先度の目安: 最優先＝影響度高×コスト低／高＝影響度高×コスト中、影響度中×コスト低／低＝影響度低。それ以外は中。
- 高影響×低コストを先頭に並べる。全部直せとは言わない（優先度なしの100件指摘は0件と同じ）。
- 同一の根本原因から生じる所見は1行に統合し、観点を併記する（例: (b)(f)）。

### STEP 4: 改善提案

- 上位の所見にBefore/Afterの改善スケッチ（コード断片）を付ける。
- その箇所を将来も守るために追加すべき契約テスト（事前条件・事後条件・不変条件の検証）を提案する。契約テストは自然言語の条件文で書けば足りる（テストコードの提示は任意）。

**✅承認ゲート**: 所見と優先度を提示し、どこまで直すかは人間が判断する。勝手に修正を実行しない（レビューと修正は別の意思決定）。

## 反証ラウンド（必須・スキップ禁止）

以下すべてに明示的に答える。該当した指摘は削除または修正し、その記録を「判断根拠」に残す。検討の結果「維持」と判断した場合も、該当の事実と維持の理由を記録する（無言スルーのみ禁止）。

1. 各指摘は一般論ではなく、**このコードの具体的な行・箇所**に遡れるか？（行番号・シンボル名のない指摘は削除）
2. 「動いているコードの趣味的な書き換え」を混ぜていないか？（変更容易性への影響を一文で言えない指摘は削除）
3. STEP 1の**目的推論が間違っている可能性**を検討したか？（目的の解釈が変わると所見も変わる。別解釈での再走査を1回行う）
4. 偽陽性チェック: フレームワーク・ライブラリの規約由来のパターン（ORMが要求するsetter、シリアライザが要求する引数なしコンストラクタ等）を設計負債と誤認していないか？

## 出力契約（この形式以外で出力しない）

1. **コードの目的推論**（ファイル・クラスごと。推論の根拠付き）
2. **所見表**: `| 箇所（行・シンボル） | 観点(a〜f) | 問題 | 影響度 | 修正コスト | 優先度 |` ＋ 観点ごとの該当あり/なし一覧
3. **上位所見のBefore/After改善スケッチ**（コード断片）
4. **追加すべき契約テスト一覧**（事前・事後・不変条件の別を明記）
5. **判断根拠・確認事項・保留事項**: 確認事項（人間への質問。非対話環境の承認ゲートはここに列挙）と保留事項（実行者が置いた仮定。目的推論の仮定を含む）を区別して書く／反証ラウンドで削除・維持した指摘の記録を含む

## アンチパターンガード

- **スタイル指摘への脱線禁止**: インデント・命名規約の揺れ・好みの書き方は対象外。対象は変更容易性のみ。
- **網羅主義の禁止**: 優先度なしの大量指摘は読まれない。高影響×低コストから絞って提示する。
- **レビューと修正の混同禁止**: 指摘と同時に勝手にコードを直さない。修正は人間の判断後、別の作業として行う。
- **機能バグ探しとの混同禁止**: ロジックの正誤・仕様との一致は別のQAで診る。
- **根拠なき一般論の禁止**: 「一般的にこうすべき」は指摘ではない。この目的のこの箇所にどう効くかを言う。

## 品質基準（Doneの定義）

- 6観点すべてに「該当あり／なし」の走査結果がある。
- すべての所見が行・シンボルに遡れ、変更容易性への影響が一文で説明されている。
- 優先度順に並び、上位所見には改善スケッチと契約テスト提案が付いている。
- 出力契約の5点が揃っている。
