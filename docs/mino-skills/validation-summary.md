# mino-skills 検証サマリー

この文書は、公開情報から再構築したmino Skillについて、GitHub上で追跡できる試行とレビューだけを使って現在の有用性を判定します。
結論は`観測済み／仮説／未確認`に分け、根拠がない成功を補いません。

## 結論

- `mino-context-discovery`は、1件のEC題材で素材に遡れる境界候補と確認事項を生成できたため、**限定的に有用性を観測済み**です。
- `mino-contract-driven-coding`は、Issueにない挙動を作らないガードを文面とレビューで確認しましたが、実装タスクとしての効果は**未確認**です。
- Skill間の参照を実在確認・明示指定に限定する運用ガードは、PRレビューで問題を検出して修正できたため**観測済み**です。
- その他のSkillは導入と静的レビューまでであり、実行時の有用性は**未確認**です。

したがって、現時点では6 Skillをlabのまま残します。core昇格や一括利用の推奨は行いません。

## 判定区分

| 区分 | 判定条件 |
|---|---|
| **観測済み** | GitHub上の入力、出力、レビュー、修正へ遡れる |
| **仮説** | 公開資料または観測結果から期待できるが、実行証拠がない |
| **未確認** | 判断に必要な入力、出力、比較対象が不足している |

## 実例から得た判断

### 残す

- **素材への遡行可能性**: `mino-context-discovery`の例では、境界ごとに根拠と保留事項を分けたことで、AIの補完と入力事実を区別できた。
- **問題別のSkill選択**: 用語境界と業務時系列を別Skillにしたことで、固定の6段パイプラインを要求せずに済む。
- **明示呼び出しと実在確認**: PR #42で未導入Skillへの案内が問題化したため、自動連鎖禁止を維持する。
- **Issue正本の優先**: PR #43で契約表が新仕様を作りうる文言を検出できたため、仕様未定時のIssue差し戻しを維持する。

### 修正する

- 検証済みという表現は、GitHub上の入力・出力へ遡れる場合だけ使う。
- 1件の成功例からSkill全体や他案件での再現性を断定しない。
- 各実行記録に、入力、期待、観測、失敗、保留、利用モデルを可能な範囲で残す。

### 見送る

- 6 Skillすべてを固定順で自動実行する運用。
- PR #33の「全Skill検証済み」という状態表記の無条件な移送。
- 実行証拠がない段階でのcore昇格や標準フローへの組み込み。

## 次の検証候補

1. **要求定義**: 既存の曖昧な機能要望に`mino-socratic-requirements`を使い、通常のChatGPT仕様化と論点・所要ターンを比較する。
2. **Skill選択**: 同じ業務素材にcontextとeventを別々に適用し、境界問題と時系列問題を混同せず分けられるか確認する。
3. **設計から契約**: 小さな既存モデルをmodel-deepeningで見直し、人間が採用した案だけをcontract-driven-codingの契約表へ渡す。
4. **変更容易性レビュー**: 既知の設計欠陥を埋めた小さなコードで、6観点の検出数、見逃し、偽陽性（問題でない箇所を問題と判定すること）を記録する。

各検証は1 Skillまたは1連携に限定し、結果が悪い場合も削除せず記録します。

## 未確認事項

- AIモデルや実行環境が変わっても同じ結果になるか。
- Skillなしの通常プロンプトより、品質または判断速度が改善するか。
- 出力の長さや確認ゲートが、実務上の負担に見合うか。
- 実案件2回以上というlab昇格条件を満たせるか。
