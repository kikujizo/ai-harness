# リスクダイヤル運用 — 「ガチガチのガードレール」から「調節可能なリスクコントロール」へ

方針: 固定的な禁止リストではなく、**不可逆性に応じて自動化の度合いを連続的に調節する**。
守るべき境界（不可逆4カテゴリ）は動かさず、それ以外のダイヤルは実績に基づいて回す。
リスクの定義そのものの正本はルートの`AGENTS.md`（本書では再定義しない）。

## 1. 動かさない境界（ダイヤルの外側・固定）

動かさない境界＝不可逆4カテゴリ（ルートの`AGENTS.md`のリスク分類参照）。判断軸は取消不能性。

+ メタループの自律化禁止（プロセス改善はL1固定・人間承認必須。`../loops/principles.md` 参照）

## 2. 回してよいダイヤル（それぞれ独立に調節可能）

各ダイヤルの**初期値は導入時に自分で決めて記入する**。以下は決め方の目安と記入欄。

| ダイヤル | 緩める方向 | 締める方向 | 初期値の決め方（記入欄） |
|---|---|---|---|
| 自動マージ条件 | 必須チェック名を最小限に | 必須チェック名を追加 | 「必須」とするチェックを列挙する。「1つ以上green」のような迂回可能な条件は避ける → `記入: ______` |
| 機械層高リスクパス | パターンを減らす | パターンを追加 | 自分のリポジトリで「自動で触られたら困るパス」を列挙する → `記入: ______` |
| fix-round上限 | 増やす（自走を長く） | 減らす（早く人間へ） | 最初は小さく（例: 1〜2回）始め、暴走コストを見てから増やす → `記入: ______` |
| 実装タイムアウト | 延ばす（大きいタスクを許容） | 縮める（早くフォールバック） | タスクの規模感に応じてフォールバック発動までの許容時間を決める → `記入: ______` |
| Issue粒度 | 大きく（1件で多く進む） | 小さく（レビュー精度優先） | 速度とレビュー精度のどちらを優先するかで決める → `記入: ______` |
| 再帰照合の適用範囲 | 品質重要時のみ | 全成果物に適用 | コスト対効果で判断する → `記入: ______` |
| 人間の事後確認頻度 | 低頻度サンプリング | 全件確認 | 信頼が積み上がるまでは高頻度から始める → `記入: ______` |

## 3. ダイヤルを回すルール（ラチェット構造）

1. **一度に1つ**だけ回す（複数同時に変えると、問題発生時にどれが原因か分からない）
2. **緩める＝証拠ベース**: 直近の運用実績で誤動作ゼロを確認してから。変更はDecision Logに記録し、
   「取り消し手順」欄を必ず埋める
3. **締める＝即時**: インシデント（不可逆4カテゴリへの接触・意図しない動作）が起きたら、
   関連ダイヤルを即1段階締める。原因分析が済むまで戻さない
4. **計測なしに回さない**: ゼロタッチ率・フォールバック発動率・fix-round頻度など、
   自分の計測ログが見えていない状態でのダイヤル操作は勘になる。まず計測、次に調節

## 4. 判断に迷ったときの1問

**「この変更が最悪の形で失敗したとき、`git revert`（または相当の操作）で完全に元に戻せるか？」**

- Yes → 通常レーン。ダイヤルの範囲で自動化してよい
- No → 不可逆。境界側。人間の承認を挟む

## 5. 効率×リスクの現在地マップ（自分で埋めるテンプレ）

導入時にどこまで自動化されているかをフェーズごとに可視化する。実績が溜まったら緩める側へ動かす。

```

```
