# タスク→レーン/AI ルーティング

最初の分岐は「どのAIか」ではなく「**どのレーンか**」。

```
タスク発生
 ├─ 開発タスクで、対象リポジトリに自動化パイプライン（CI/CD等）が導入済み → 【自動レーン】起票して手を離す
 ├─ 開発タスクだが、パイプライン未導入 or パイプライン自体の変更 → 【対話レーン】
 └─ 開発以外（文章・分析・整理・壁打ち） → 【対話レーン】
```

## 自動レーン（自動化パイプラインがある場合。以降ゼロタッチ）

| 人間がやること | それだけ |
|---|---|
| 意図（要望）を入力する—要件化・起票（Issue等）はAIが行う | パイプラインがPM評価→実装→レビュー→マージまで進める |
| 高リスク時のみ: 不可逆操作の発効点（merge等）でのapprove/deny | 不可逆4カテゴリだけ人間が承認する運用にする（`docs/risk-dial.md`） |
| 停止・エラー時: 原因確認→やり直し | 通知の仕組みは自分の環境（GitHub通知・チャット連携等）に合わせて用意する |

**自動レーンに乗せないもの**: パイプライン自己変更（高リスクカテゴリ③。AIが実装し、
実装AIと独立したレビュー＋発効点で人間approve→AIがmerge実行を必須）、仕様が固まっていないもの（先に仕様化担当で要件化）。

## 対話レーン（人間が各AIと直接やりとり）

以下は4AI体制を前提にした**一例**。実際の担当ツール・枠の逼迫状況に応じて、
このレーン表は**自分の環境の実情に合わせて書き換える**。

| タスク | 一次担当 | 備考 |
|---|---|---|
| 「作るべきか」から考えたい・壁打ち | 仕様化担当（ChatGPT等） | 無制限枠なら遠慮なく回す |
| 要件定義→Issue化 | 仕様化担当 | 出力はIssueテンプレ準拠の本文 |
| パイプライン自体の変更・設計相談 | 技術スーパーバイザ（Claude Code等） | 設計チェック（`.agents/skills/design-check/SKILL.md`）→AI実装（独立レビュー＋発効点で人間approve必須） |
| 原因不明のエラー・環境トラブル | 技術スーパーバイザ | 対話で調査。2敗ルールは対話レーンでも有効 |
| ファイル雑務（整理・変換・一括処理） | 実装担当（軽量）/ 技術スーパーバイザ（複雑） | 破壊系は事前にdry-run |
| 文章生成（記事・報告・要約） | 実装担当 / 技術スーパーバイザ | 品質重要なら `.agents/skills/recursive-writing/SKILL.md` |
| 対話でのIssue評価・技術相談 | 技術PM（Codex等） | 枠が逼迫していれば短いコンテキストの仕事だけ渡す |
| 調査・DeepResearch系 | 仕様化担当 | 結果を基準ファイル化すると再利用できる |

## 枠の配分原則

**逼迫している枠ほど、短いコンテキストの仕事に割り当てる。** 以下は4AI体制の場合の例（自分の枠状況に合わせて書き換える）:

```
仕様化担当（枠が広い想定）   → 人間対話・壁打ち・要件化（長くてよい）
技術PM（枠が逼迫しがち）    → PM判断のみ（Issue本文だけ読む短い仕事）
実装担当（枠に余裕がある想定）→ 実装の物量
技術スーパーバイザ（枠に余裕）→ 品質（レビュー）と例外（フォールバック・スポット）
```

**Claude Code枠の内側**にもモデル配分がある: 指揮者（最上位モデル）は設計・検証・統合・提案に専念し、
大量・機械的処理は軽量モデル、中規模の独立サブタスクは中位モデルのサブエージェントへ委譲する。
詳細は [orchestration.md](./orchestration.md)（Claude Codeの既定挙動）。

## 選択の原則（対話レーン共通）

1. 2回失敗したら粘らず移管（リトライの積み重ねが最大のトークン浪費）
2. 同じ依頼を2度説明していたら仕組み化のサイン（Skills化・基準ファイル化・自動レーンへの昇格を検討。
   `.agents/skills/knowledge-reflux/SKILL.md` 参照）
3. 迷ったら「不可逆か？」で判断（`docs/risk-dial.md` §4の1問）
