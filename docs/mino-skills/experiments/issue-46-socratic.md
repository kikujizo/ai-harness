### 質問1

- 問い: 効率化したいのは、誰が・何を確認するレビューですか？（例：ChatGPTによるIssueの要件確認、CodexによるPRの技術レビュー。別の組み合わせでも構いません）

### 質問2

- 問い: 「効率化できた」と判断するうえで、最も重要な変化はどれですか？（例：5時間枠内に完了できるレビュー件数、1件のレビューを中断せず完了できる割合、1件あたりのトークン消費量）

- 問い: 「1件のレビュー」は、どこからどこまでを1件として数えますか？（A. 最初のレビュー結果を出すまで／B. 修正後の再レビューを含め、最終判断を出すまでの一連のやり取り／C. その他の範囲）
- 回答: 「B」。
- 得られた発見: 1件の境界は、初回レビューから修正後の再レビューを経て最終判断を出すまで。

### 質問4

- 問い: 一連のレビューの中で、トークンを最も多く消費していると感じる場面はどれですか？（A. 前提情報の読み込み／B. 初回のレビュー判断／C. 修正後の再読・再レビュー／D. 説明・引き継ぎ／E. 未特定・その他）

### 質問5

- 問い: 1件あたりのトークン消費量が減っても、レビュー結果がどのような状態なら「効率化できた」とは言えませんか？
- 回答: 「その工程で見つからないミスがあとから発覚して、追加のトークンが必要な場合」。
- 得られた発見: 短期的な削減だけでは不十分。見落としによる後工程の手戻りと追加消費を含めて判断する必要がある。

### 質問6

- 問い: 最終判断後に発覚する「そのレビューで見つけるべきだったミス」は、1件のレビューあたり何件までなら品質が維持されたと判断できますか？

- 得られた発見: 見落とし0件は目標値。合否は効率化前後の見落とし発生頻度を比較し、効率化後に増えた場合を不合格とする。

### 質問7

- 問い: 現在の「1件あたり平均トークン消費量」は、効率化前の比較基準としてすでに計測されていますか？
- 回答: 「されていません」。
- 得られた発見: 現状値は未計測で、効率化前後を比較する基準線がない。

### 質問8

- 問い: 基準線を得るための計測に、レビュー1件あたりどの程度の追加負担まで許容できますか？（例：追加作業なし、数分程度、レビュー後にまとめて記録できる範囲）

- 得られた発見: レビュー中の追加記録ではなく、レビュー後に一括記録できる範囲の負担なら許容される。

### 質問9

- 問い: 最初に基準線を測る対象は、どれを優先しますか？（A. 実装前のIssue評価／B. 実装後のPRレビュー／C. 両者を別々に測る／D. 優先順位なし・その他）

- 得られた発見: 最初の測定対象は実装後のPRレビュー。Issue評価は今回の第一対象ではない。

### 質問10

- 問い: 1件あたりのトークン消費量を減らす以外に、Codexの5時間リミットが先に来る問題を軽減できる方法として、何が考えられますか？

## 3. 要求定義書

### 目的

Codexの5時間枠に対する実装後PRレビューの消費実態を、改善効果を相殺しない計測負担で比較可能にし、後続の効率化判断を根拠のあるものにする。

### 背景

- Codexは主にレビュー用途で使われ、他AIより5時間リミットが先に来ると認識されている（質問1）。
- 最重要指標は平均的なレビュー1件あたりのトークン消費量である（質問2）。
- 1件は初回レビューから修正後の再レビューを経て最終判断に至るまでを含む（質問3）。
- 最大消費箇所は未特定で、資料読み込みが大きいという見方も未確認の推測である（質問4）。
- 現状値が未計測で、比較基準線がない（質問7）。
- 計測負担はレビュー後にまとめて記録できる範囲に限る（質問8）。
- 最初の対象は実装後PRレビューである（質問9）。

### 解くべき問題

AIハーネス運用者が、計測自体のトークン負担で改善効果を食い潰さない範囲で、実装後PRレビュー1件あたりの平均トークン消費量を把握し、効率化前後の差を比較判断できる基準線を持てていない問題。

### スコープ外

- 実装前のIssue評価に対する基準線計測（質問9により後順位）。
- トークン消費の最大箇所を未検証のまま資料読み込みと断定すること（質問4）。
- トークン削減策そのものの設計・実装・評価。
- 修正と再レビューのラリー回数削減（質問10に基づく次段の別問題）。
- Codexの5時間制限そのものの変更。

### 成功基準

1. 対象となる実装後PRレビューについて、初回レビュー開始から最終判断までを1件として識別できる。
2. 測定対象を `N` 件としたとき、各レビューのトークン消費量が数値で記録され、`合計トークン消費量 ÷ N` により基準線となる平均値を算出できる。
3. レビュー中の計測用追加記録は0回で、必要な記録は各レビューの終了後にまとめて行われる。
4. 計測自体に要した追加トークン量または追加作業時間を数値で記録し、後続の改善検証で得られた削減量を計測負担が上回っていないか判定できる。
5. 品質確認では、効率化前後の「レビューで見つけるべきだったミス」の発生頻度を同じ定義で算出し、効率化後に増えていないことを合格条件とする。0件は目標値だが、単発1件を直ちに不合格とする基準ではない。

### 未確認の前提

- Codexのトークン消費量をレビュー1件単位で取得できる、または妥当な方法で対応付けられること。
- 異なる重さのPRを比較可能にする分類軸と、最初に測定する件数 `N`。
- 計測自体の追加トークン量または追加作業時間を取得できること。
- 「レビューで見つけるべきだったミス」の対象範囲と、発覚を追跡する期間。
- トークン消費量の具体的な削減目標値。現段階では基準線がないため未設定。

## 4. 判断根拠・確認事項・保留事項

### 判断根拠

- 人間は問題候補のうち、未計測で比較基準線がない問題を今回の主問題として選択し、レビュー往復回数の問題は次段へ分離した。
- 「計測自体にトークンが必要なら本末転倒」との補足を受け、計測負担が改善効果を食い潰さないことを主問題と成功基準へ反映した。
- 最終問題文は人間が「いいですね」と明示承認した。

### 確認事項（人間への質問）

- レビュー1件単位のトークン消費量を、現在のCodex環境から取得できるか。
- 比較可能なPRの重さを区分する軸と、最初の測定件数 `N` をどう定めるか。
- 見落とし発生頻度の対象となるミスの範囲と追跡期間をどう定めるか。
- 基準線取得後に、平均トークン消費量の削減目標をどの値にするか。

### 保留事項（実行者が置いた仮定）

- 「AIハーネス運用者」を、この問題の判断と計測結果を利用するアクターとして置いた。個人・チームなどの詳細は未特定。
- 成功基準では、トークン量を取得できない可能性を残し、取得可否そのものを確認事項とした。取得可能とは仮定していない。
- 次段の「ラリー回数削減」は候補として記録するだけで、今回の計測方法や解決策には含めていない。

### 反証ラウンド（5項目）

1. **問題文が手段になっていないか**: 問題文は「計測機能を作る」「仕組みを導入する」ではなく、比較判断できる基準線を持てていない状態を記述しているため、手段への固定はない。修正せず維持した。
2. **成功基準が目的の言葉で書かれているか**: 基準線の算出、計測負担との比較、品質頻度の非増加という目的上の達成状態で記載した。特定ツールの導入を完了条件にしていないため、修正せず維持した。
3. **「誰の」問題かが特定されているか**: 最終問題文に「AIハーネス運用者」を明記した。運用者の詳細な単位は未確認のため保留事項にも記録した。
4. **「何もしない」選択肢と比較したか**: 何もしなければ現状値と比較基準線がないままで、効率化の効果や計測負担が効果を相殺したかを判断できない。5時間枠の圧迫に対する改善判断も根拠を持てないため、主問題を維持した。
5. **問答ログに誘導尋問がないか**: 10問を確認した。例示・選択肢は回答形式を具体化するためのもので、結論を回答内に埋め込んだ問いはない。質問6の「0件」は人間の回答後に目標値と合否基準を分けて再確認し、前後頻度の非増加へ修正した。

### 人間負担・運用上の観測

- 最初の試行は、実験の背景と目的が人間へ十分説明されないまま質問が始まり、質問意図が伝わらず中断した。
- 人間の同意後に背景説明を行って最初から再開始し、質問数を0へ戻した。中断前の回答は有効な実験回答として採用せず、本問答ログ・問題定義・10問の質問数から除外した。
- この中断は要求内容の証拠には使わず、「対話開始前に目的と回答者の役割を共有しないと、人間負担と混乱が増える」という運用上の観測としてのみ記録する。
