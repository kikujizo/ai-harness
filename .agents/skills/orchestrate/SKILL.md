# orchestrate: 指揮者として委譲を設計する

reframe-question で問いを正した後、実行体制をここで設計する。
lateral-sweep の大量走査など、機械作業はここを通して下位モデルへ流す。

## 手順

1. タスク開始時に分解する。**判断が要る部分**（指揮者=最上位モデルが保持: 統合・採否判断・
   ユーザー報告）と**機械的・大量・独立な部分**（下位モデルへ委譲）に分け、何をどのモデルで
   やるかを最初に設計する（委譲基準とティアの正本: `docs/harness/ops/orchestration.md`）
2. 独立なサブタスクは**並列に投げる**（直列にしない）
3. 委譲プロンプトは自己完結で書く。相手はこのスレッドを見ていない前提で、背景・対象パス・
   期待する報告形式・read-only等の制約を明記し、「あなたの報告だけを根拠に判断される」と伝える
4. **1〜2ファイル読んで即答できる軽作業は委譲しない**（委譲の方が高くつく。
   `docs/harness/ops/token-discipline.md`）
5. サブエージェントの報告のうち重要な主張・完了報告は、一次資料（コミットSHA・PR・実ファイル）で
   実在検証してから採用する（`docs/harness/loops/failure-map.md`「幻の完了報告」）
6. 自分の能力で詰まったら、上位モデルへの質問・委譲を**ユーザー承認の範囲で**エスカレーションする

実例: レビュー指摘2件への対応で、原本37ファイルの機械点検をSonnetへ、事実確認をSonnetへ並列委譲し、
指揮者は分類・採否判断・還流設計のみ実施→検出18件・幻報告1件を検出した。

## 制約

- 委譲中のエージェントと同じファイルを並行して触らない
- サブエージェントの結論を無検証のまま最終報告・GitHubコメントに載せない
- 委譲しても最終責任は指揮者（報告の品質チェックを省かない）
- 日本語で出力（コード・識別子は原語）
