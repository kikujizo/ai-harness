# 流用OSSカタログ（車輪の再発明をしないための基準）

方針: ゼロから作らない。ただし何でも入れてよいわけではない。**導入前チェック**を先に固定し、
具体的な採用リストはあくまで時点情報の付録として扱う。

## 導入前チェック観点（本体）

新しいOSS・外部ツール・Skillを導入する前に、必ず次を確認する:

1. **提供元** — 信頼できる主体か（個人の野良リポジトリか、組織・企業が継続メンテしているか）
2. **更新日** — 最終更新が古すぎないか。放置されたプロジェクトに依存しない
3. **要求権限** — 何にアクセスするか（ファイルシステム全体か、特定ディレクトリのみか、
   外部送信はあるか）。最小権限で足りるものを選ぶ
4. **Skill系は導入前にスキャンする** — 実行前にセキュリティスキャン用ツールを通してから入れる
   （プロンプトインジェクションや不要な外部通信が仕込まれていないか）

**判断基準の使い方**: 「Star済み」「有名だから」だけで採否を決めない。上記4点を毎回確認してから
入れる。必要になるまで先回りして入れない（依存が増えるほど§トークン規律の固定費が増える）。

## 付録: 例（時点情報・各自更新）

以下は執筆時点で参考にしていたOSS・情報源の例。バージョン・URL・提供状況は変わるため、
導入前に必ず最新情報を上記チェック観点で確認し直すこと。ここに載っているという事実は推奨を意味しない。

### 導入を検討してよいもの（例）

| OSS | 用途 | 備考 |
|---|---|---|
| Claude Code向けの安全ガードレール集（secret検知pre-commit、main保護、`--no-verify`禁止、監査ログ等） | 「ルールに書いても守らない」を権限とフックで塞ぐ | フック部分だけ流用する運用が現実的 |
| ツール出力・ログの圧縮ツール | トークン消費を大きく削減 | まず非重要プロジェクトで1週間試し、精度劣化がないか確認してから本採用 |
| Skillのセキュリティスキャンツール | 導入するSkillの安全性確認 | 外部Skill導入時の関所として使う |

### 必要になったら検討するもの（例）

| OSS | 用途 | 判断基準 |
|---|---|---|
| 仕様駆動開発のCLI+テンプレ一式 | 要件定義・Issue運用の型化 | 仕様化→Issue運用の粒度が安定しなくなったら検討 |
| 公式Skills集（文書生成、skill-creator等） | 定型文書生成の自動化 | 業務で文書生成が必要になったら |
| TDD・計画・サブエージェント実行の大型ワークフローフレームワーク | 本格的なプロダクト開発の型 | 個人・小規模開発には過剰なことが多い。入れるなら一部機能だけ選択導入 |
| 技術スタック別のエージェント向けルール集 | 新しい言語・フレームワーク導入時の参考 | 丸ごとコピーせず短く削って使う |
| UIデザイン基準集 | UI実装タスクの基準ファイル | UIを作るタスクが来たら、`docs/criteria/` の基準ファイルとして流用を検討 |

### 参考にした情報源（例）

- AGENTS.md公式ガイド系の解説 — サイズ上限・簡潔さの重要性
- AGENTS.md仕様の比較解説 — クロスツール標準としての位置づけ
- 各AIツール公式のAgent運用ベストプラクティス — ルールは要点のみ・ファイル参照で薄く
- Rules/Skills/Commandsの使い分け解説
- Spec-Driven Developmentの解説記事 — 仕様先行の経済性
