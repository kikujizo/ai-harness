# loop-design: ハードガード付きでループを設計する

新しいループ（自動レーン・定期実行・リトライ・エージェントループ）は、
このスキルの5ステップを全通過してから動かす。答えられない項目があるループは動かさない。
背景原則は `docs/harness/loops/principles.md`、台帳形式は `docs/loop-ledger.md`。

## 手順

### ステップ1: 6構成要素を埋める（`docs/harness/loops/principles.md` §1）

次の6つをすべて言語化する。埋まらない要素があるループは設計未了。

1. **自動化トリガー** — 何が起動するか
2. **並列隔離** — 作業空間をどう分離するか
3. **知識バンドル** — 反復間で何の知識を持ち回るか
4. **外部連携** — どのツールに接続するか
5. **分業設計（Maker-Checker）** — 生成役と検証役をどう分けるか
6. **状態の外部化** — 現在位置をどこに永続化するか

### ステップ2: 設計チェックリスト10項を全通過させる

1つでも×があるループは動かさない。

1. **目的関数** — 「成功した」と機械判定できる条件は何か（テスト・verdict・基準ファイル）。人間の感想でしか判定できないなら、それはループではなく対話タスク。
2. **外部信号** — 反復ごとに何が外部から入るか（テスト結果・レビュー・基準照合・実行ログ）。純粋な自己内省だけのループは収束しない。
3. **反復上限** — 何回で打ち切るか（数値で）。
4. **時間上限** — 1反復・全体それぞれの timeout（数値で）。
5. **無進展検出** — 「前の反復と同じ結果」をどう検知して止めるか。
6. **予算** — 1周あたりの想定トークン・枠消費と、どの枠を使うか。逼迫している枠ほど短いコンテキストの仕事に割り当てる。
7. **停止時の遷移先** — 上限到達・失敗時に誰に・何が・どう通知されるか（自動停止 → 人間の経路があるか）。
8. **状態の外部化** — 再起動後も現在位置が分かるか（ラベル・ファイル。メモリ内だけの状態は禁止）。
9. **分業** — 生成役と検証役は別か（同一モデルの自己採点になっていないか）。
10. **自律度と昇格計画** — L1/L2/L3 のどれで始め、何の実績があれば上げるか。下げるトリガー（インシデント定義）は何か。

### ステップ3: 4ハードガードを数値で決める（`docs/harness/loops/principles.md` §2）

チェックリスト3〜6を、具体的な数値として確定する。

- **反復上限**: ___ 回
- **時間上限**: 1反復 ___ / 全体 ___
- **無進展検出**: 「___ が ___ ターン変化しなければ停止」
- **予算キャップ**: 1周 ___ トークン相当 / どの枠 ___

### ステップ4: 自律度は L1 から開始する

新しいループは必ず **L1（報告のみ）** で始める。昇格は「直近10回で誤動作ゼロ + 復旧手順の文書化」を根拠に1段階ずつ。
インシデント（不可逆4カテゴリに触れる／意図しない不可逆操作）1回で即1段階降格。
不可逆4カテゴリの定義はルートの`AGENTS.md`（リスク分類）が正本。
ループがループ自身やルールを書き換えるメタループは**恒久L1**。

### ステップ5: 台帳エントリを作る（`docs/loop-ledger.md`）

7項目（トリガー / 状態 / 上限 / 停止条件 / 自律度 / 計測 / ギャップ）を埋めて台帳に登録する。
**台帳にないループは存在してはならない。** 昇格・降格の変更は Decision Log（`docs/templates.md`）に記録し、「取り消し手順」欄を必ず埋める。

## トークンガード

- 既存の似たループがあれば、ゼロから設計せず台帳エントリを複製して差分だけ埋める。
- このスキルは**新設・自律度変更のときだけ**使う。既存ループの1回の実行には使わない。
