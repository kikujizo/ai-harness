# assessment-first: 修正の前に評価報告を出す

指摘や提案には、すぐ手を動かさず「実行前評価報告」で返す
（オーナーが 2026-07-08 に「次回からこういった提案が欲しい」と明示承認した型。
GitHub 上の確認 URL は Decision Log で未特定のため、merge 時は lab として扱う）。

## 手順

1. **TLDR**: 全件妥当か / 採用しない提案はどれか / レビュアーが見落とした追加発見はあるか
2. 指摘ごとに3点セットを出す（原理の正本は `docs/harness/ops/orchestration.md` §4 —
   「指摘が正しいこと」と「修正案が正しいこと」は別々に検証する）:
   - **妥当性評価**: その指摘が正しいか
   - **分類**: 根本（正本・上流起因）か 対象依存か。判定質問は「ゼロから新規に構築・適用しても起きるか？」
   - **具体的対応内容**: `recursive-review` 手順4と同じディスポジション三択で宣言する —
     **今回修正 / wontfix（理由記録） / 後回し（追跡Issue URL必須）**。
     レビュアーの修正案が悪手なら従わず別解と理由を示す
     （例: 正本の分岐を生む修正案は参照化で置き換える）
3. **横展開チェック**（条件付き）: 次のいずれかに該当する場合のみ `lateral-sweep` を呼ぶ:
   - 正本起因の失敗クラスが疑われる
   - 同種ミスが複数箇所に広がる可能性がある
   - セキュリティ・権限・データ損失に触れる
   - 人間またはレビュアーが横展開を明示した
   該当しない場合は「横展開不要」と明記し、`lateral-sweep` を呼ばない
4. **還流判定まとめ**: 正本側の修正が必要なものを一覧化する（`docs/harness/knowledge/reflux.md`）
5. 実行プランを提示する。
   - **人間が対象指摘・実装担当・実行許可を既に明示している場合**: 二重承認を求めず実行に進んでよい
   - **担当不明・許可不明の場合**: 停止し、GOまたは担当指定を求める

## 出力

TLDR → 指摘ごとの3点セット → 横展開の要否と結果 → 還流判定 → 実行プラン。
評価と対応はGitHubのIssue/PRへコメント記録する（ルート`AGENTS.md`「GitHubドリヴン記録」）。

## 制約

- 評価報告なしに修正を始めない
- 人間が「対応して」「実装して」と担当・許可を明示済みなのに、再度GOを要求しない
- 他AIの「対応済み」報告は成果物の実在（コミットSHA・PR番号）を一次資料で検証してから信用する
  （`docs/harness/loops/failure-map.md`「幻の完了報告」）
- 日本語で出力（コード・識別子は原語）
